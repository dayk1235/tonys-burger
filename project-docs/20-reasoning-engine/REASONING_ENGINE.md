## Preamble

This document is the canonical specification of Reasoning inside Restaurant OS.

It is not a knowledge document. It is not a reference manual. It is a **constitutional engine** — the specification of how Restaurant OS transforms validated cognition into understanding, explanation, and recommendation.

Everything before this — Observation, Pattern, Evidence, Memory, Knowledge, Attention — exists only to serve Reasoning. Everything after this — Decision, Action, Learning — depends on Reasoning.

Reasoning is where intelligence begins.

## Part 1 — Philosophy of Reasoning

### 1.1 What Reasoning Is

Reasoning is the cognitive process by which Restaurant OS transforms information into understanding.

Reasoning takes what the system knows (Knowledge), what it has experienced (Memory), what it perceives (Observation), what it recognizes (Pattern), what it accepts as valid (Evidence), and what it considers important (Attention) — and produces explanations, recommendations, and decision support.

Reasoning does not create facts. Reasoning does not replace Knowledge. Reasoning does not execute action. Reasoning is the constitutional bridge between Attention and Decision.

### 1.2 Why Reasoning Exists

Reasoning exists because information without understanding is noise.

Observation detects. Pattern recognizes. Evidence validates. Memory retains. Knowledge accumulates. Attention selects. None of these produce understanding. Only Reasoning transforms raw cognitive inputs into actionable comprehension.

Without Reasoning, Restaurant OS would be a perfect recording machine — capturing everything, understanding nothing.

### 1.3 Distinctions

Reasoning is different from the following cognitive entities.

#### Reasoning vs Observation

| Property | Observation | Reasoning |
|----------|-------------|-----------|
| **What it is** | The smallest unit of perception. A raw signal captured from the environment. | The transformation of observations into understanding. |
| **What it produces** | A recorded perception. | An explanation, recommendation, or decision support. |
| **Cognitive role** | Input. Reasoning consumes observations. | Processor. Reasoning transforms observations. |
| **Can it be wrong?** | Yes — false observation. | Yes — flawed reasoning. |
| **Error mode** | Perceptual error. | Reasoning error. |

**Why they are different:** Observation captures what is. Reasoning determines what it means.

#### Reasoning vs Pattern

| Property | Pattern | Reasoning |
|----------|---------|-----------|
| **What it is** | A recognized repetition across observations. The system sees the same thing happening. | The cognitive process that uses patterns as inputs. |
| **What it produces** | A pattern statement: "X happens when Y." | An explanation: "X happens because of Y, therefore we should Z." |
| **Cognitive role** | Pattern is a reference. It informs reasoning. | Reasoning uses patterns as evidence. |
| **Can it be wrong?** | Yes — false pattern. | Yes — reasoning may misinterpret patterns. |

**Why they are different:** Pattern recognizes repetition. Reasoning understands causation.

#### Reasoning vs Evidence

| Property | Evidence | Reasoning |
|----------|----------|-----------|
| **What it is** | Validated information that supports or contradicts a claim. | The cognitive process that evaluates evidence and draws conclusions. |
| **What it produces** | A justification record. | An explained conclusion. |
| **Cognitive role** | Evidence is an input to reasoning. | Reasoning evaluates evidence. |
| **Best question** | "What supports this?" | "What does this mean?" |

**Why they are different:** Evidence is the raw material of justification. Reasoning is the process that uses it.

#### Reasoning vs Memory

| Property | Memory | Reasoning |
|----------|--------|-----------|
| **What it is** | The retained experience of the system. Past events stored for future reference. | The cognitive process that uses memory as context. |
| **What it produces** | A recall of past events. | A current understanding informed by past events. |
| **Cognitive role** | Memory provides context. | Reasoning applies context. |
| **Error mode** | Forgetting or distortion. | Misapplication of memory. |

**Why they are different:** Memory remembers what happened. Reasoning determines whether it matters now.

#### Reasoning vs Knowledge

| Property | Knowledge | Reasoning |
|----------|-----------|-----------|
| **What it is** | Validated understanding that has survived contradiction. Earned certainty. | The process that applies knowledge to new situations. |
| **What it produces** | A truth statement. | An applied understanding. |
| **Cognitive role** | Knowledge is a permanent resource. | Reasoning is a transient process. |
| **Constraint** | Knowledge cannot be created by reasoning alone. | Reasoning cannot create knowledge without validation. |
| **Relationship** | Knowledge constrains reasoning. Valid knowledge limits what reasoning can conclude. | Reasoning extends knowledge. Reasoning applies knowledge to new cases. |

**Why they are different:** Knowledge is what the system knows. Reasoning is what the system does with what it knows.

#### Reasoning vs Attention

| Property | Attention | Reasoning |
|----------|-----------|-----------|
| **What it is** | The allocation of cognitive resources to stimuli. The scarcest resource. | The cognitive process that uses allocated attention. |
| **What it produces** | A prioritized queue of stimuli to process. | Understanding from the attended stimulus. |
| **Cognitive role** | Attention selects what matters. | Reasoning processes what matters. |
| **Flow** | Attention precedes reasoning. Every reasoning process begins with an attention allocation. | Reasoning follows attention. Without attention, no reasoning occurs. |

**Why they are different:** Attention decides what to think about. Reasoning does the thinking.

#### Reasoning vs Learning

| Property | Learning | Reasoning |
|----------|----------|-----------|
| **What it is** | The process of extracting new knowledge from experience. | The process of applying existing knowledge to new situations. |
| **What it produces** | New knowledge, pattern updates, memory refinement. | Explanations, recommendations, decision support. |
| **Cognitive role** | Learning improves the system over time. | Reasoning serves the system in the moment. |
| **Flow** | Reasoning may trigger learning (when reasoning discovers a gap). | Learning feeds reasoning (better knowledge improves reasoning). |

**Why they are different:** Learning improves what the system knows. Reasoning uses what the system knows to understand the present.

#### Reasoning vs Decision

| Property | Decision | Reasoning |
|----------|----------|-----------|
| **What it is** | The selection of a course of action from alternatives. | The process that builds the alternatives and evaluates them. |
| **What it produces** | A commitment to act. | The analysis that makes action possible. |
| **Cognitive role** | Decision ends deliberation. | Reasoning enables deliberation. |
| **Error mode** | Wrong choice. | Flawed analysis leading to wrong choice. |
| **Boundary** | Reasoning stops where decision begins. Reasoning prepares; decision executes. | Reasoning does not decide. Reasoning prepares decisions. |

**Why they are different:** Reasoning prepares. Decision commits.

#### Reasoning vs Planning

| Property | Planning | Reasoning |
|----------|----------|-----------|
| **What it is** | The construction of a sequence of actions to achieve a goal. | The analysis that informs the plan. |
| **What it produces** | A plan — ordered actions with timeline. | The understanding that makes planning possible. |
| **Cognitive role** | Planning extends reasoning into action sequences. | Reasoning provides the foundation for planning. |
| **Flow** | Reasoning → Decision → Planning | Reasoning precedes planning. Planning cannot happen without understanding. |

**Why they are different:** Reasoning understands the situation. Planning determines what to do about it.

#### Reasoning vs Prediction

| Property | Prediction | Reasoning |
|----------|-----------|------------|
| **What it is** | A forecast of future states based on patterns and knowledge. | The analysis that generates the prediction. |
| **What it produces** | A forecast. | The reasoning chain behind the forecast. |
| **Cognitive role** | Prediction is an output. | Reasoning is the process. |
| **Relationship** | Reasoning evaluates prediction confidence and conditions. | Every prediction should be supported by reasoning. |

**Why they are different:** Prediction says what will happen. Reasoning explains why.

#### Reasoning vs Reflection

| Property | Reflection | Reasoning |
|----------|-----------|------------|
| **What it is** | Reasoning about reasoning. Meta-cognitive evaluation. | First-order reasoning about the business. |
| **What it produces** | Assessment of reasoning quality. | Assessment of business situations. |
| **Cognitive role** | Reflection audits reasoning. | Reasoning operates on business cognition. |
| **Level** | Meta-cognitive. | Cognitive. |

**Why they are different:** Reasoning thinks about the business. Reflection thinks about reasoning.

#### Reasoning vs Explanation

| Property | Explanation | Reasoning |
|----------|-------------|-----------|
| **What it is** | A communication of understanding to the Owner. | The cognitive process that builds the understanding. |
| **What it produces** | An communicated narrative. | An internal understanding. |
| **Cognitive role** | Explanation is an output of reasoning. | Reasoning is the process that produces the explanation. |
| **Audience** | Owner-facing. | System-internal. |

**Why they are different:** Reasoning builds understanding. Explanation communicates it.

#### Reasoning vs Understanding

| Property | Understanding | Reasoning |
|----------|--------------|-----------|
| **What it is** | The cognitive state of comprehending a situation. | The process that produces that state. |
| **What it produces** | A state of knowing why. | The transformation of information into comprehension. |
| **Cognitive role** | Understanding is the goal. | Reasoning is the path to the goal. |
| **Relationship** | Understanding is the product of reasoning. | Reasoning exists to produce understanding. |

**Why they are different:** Understanding is the destination. Reasoning is the journey.

#### Reasoning vs Intelligence

| Property | Intelligence | Reasoning |
|----------|-------------|-----------|
| **What it is** | The capacity of the system to achieve goals through cognition. | A specific cognitive function within intelligence. |
| **What it produces** | Goal achievement across domains. | Understanding in specific situations. |
| **Cognitive role** | Intelligence is the overall capability. | Reasoning is a component of intelligence. |
| **Scope** | Intelligence is broad. | Reasoning is focused. |

**Why they are different:** Intelligence is the system's overall cognitive capability. Reasoning is one of the functions that intelligence uses.

#### Reasoning vs Truth

| Property | Truth | Reasoning |
|----------|-------|-----------|
| **What it is** | The correspondence between cognition and reality. | The process of approximating truth. |
| **What it produces** | A correct representation of reality. | An approximation that may or may not correspond to reality. |
| **Cognitive role** | Truth is the standard. | Reasoning is the approximation process. |
| **Limitation** | Reasoning does not guarantee truth. | Reasoning can be wrong while being logically sound (valid inputs → correct logic → wrong if input is false). |

**Why they are different:** Truth is the property of correct understanding. Reasoning is the process that seeks it.

#### Reasoning vs Logic

| Property | Logic | Reasoning |
|----------|-------|-----------|
| **What it is** | The formal structure of valid inference. Rules that preserve truth. | The cognitive application of logic to business situations. |
| **What it produces** | Formally valid conclusions. | Business-relevant understanding. |
| **Cognitive role** | Logic is a tool of reasoning. | Reasoning uses logic as one of its tools. |
| **Boundary** | Logic is necessary but not sufficient. | Reasoning includes logic plus context, judgment, experience. |

**Why they are different:** Logic is formal and content-independent. Reasoning is contextual and content-dependent.

#### Reasoning vs Inference

| Property | Inference | Reasoning |
|----------|-----------|-----------|
| **What it is** | The logical step from premises to conclusion. | The complete cognitive process that includes inference. |
| **What it produces** | A derived statement. | A complete understanding. |
| **Cognitive role** | Inference is a component of reasoning. | Reasoning includes inference plus context, evaluation, judgment. |
| **Scope** | Inference is one step. | Reasoning is the full cognitive process. |

**Why they are different:** Inference draws a conclusion. Reasoning builds understanding through multiple inferences, evaluations, and judgments.

#### Reasoning vs Judgment

| Property | Judgment | Reasoning |
|----------|----------|-----------|
| **What it is** | The evaluation of alternatives against criteria. | The cognitive process that produces judgment. |
| **What it produces** | A valuation or ranking. | The understanding that enables valuation. |
| **Cognitive role** | Judgment is a component of reasoning. | Reasoning produces the inputs for judgment. |
| **Relationship** | Reasoning without judgment is incomplete. | Judgment without reasoning is arbitrary. |

**Why they are different:** Judgment evaluates. Reasoning builds the understanding that makes evaluation meaningful.

### 1.4 Philosophical Principles

#### Principle 1: Reasoning is the Bridge

Attention selects. Reasoning transforms. Decision commits. Reasoning exists between Attention and Decision — and it exists for no other purpose. Every reasoning process begins with an Attention allocation and ends with a Decision preparation.

#### Principle 2: Reasoning Never Invents Facts

Reasoning does not create knowledge. Reasoning does not generate evidence. Reasoning does not produce observations. Reasoning works with what it is given. If the inputs are incomplete, reasoning must acknowledge its limitations — not fabricate missing information.

#### Principle 3: Reasoning Never Replaces Attention

Attention decides what matters. Reasoning processes what matters. Reasoning cannot override Attention's selection. If Attention says a stimulus does not matter, reasoning does not process it. Reasoning operates within Attention's allocation.

#### Principle 4: Reasoning Never Executes Actions

Reasoning prepares decisions. Decisions lead to actions. Reasoning does not act. Reasoning that attempts to act directly bypasses the constitutional boundary between cognition and execution.

#### Principle 5: Reasoning is Provisional

All reasoning is provisional. New evidence, new knowledge, or new context can invalidate previous reasoning. Reasoning is not permanent. It is a process, not a product. The product of reasoning is understanding — and understanding can be revised.

#### Principle 6: Reasoning Must Be Explainable

If the system cannot explain how it reached its conclusion, it has not reasoned — it has guessed. Reasoning must produce an explanation chain that can be communicated to the Owner. Unexplained reasoning is not reasoning.

#### Principle 7: Reasoning Acknowledges Uncertainty

Reasoning must know what it does not know. Certainty is earned, not assumed. Reasoning must estimate its confidence and communicate uncertainty alongside conclusions. Reasoning that presents uncertainty as certainty is not reasoning — it is pretending.

#### Principle 8: Reasoning is Constrained by Knowledge

Valid Knowledge constrains reasoning. If Knowledge says X is true, reasoning cannot conclude not-X without contradicting Knowledge. Contradictions must be explicitly resolved, not silently ignored.

#### Principle 9: Reasoning Respects Evidence

Reasoning must weigh evidence. It cannot ignore evidence that contradicts its conclusion. Cherry-picking evidence is not reasoning — it is rationalization.

#### Principle 10: Reasoning is For the Business

Every reasoning process must serve business outcomes. Reasoning for its own sake is cognitive waste. The measure of good reasoning is whether it produces better business understanding.

## Part 2 — Reasoning Lifecycle

### 2.1 Lifecycle Overview

Every reasoning process progresses through a defined lifecycle. The lifecycle ensures that reasoning is complete, traceable, and auditable.

### 2.2 Lifecycle Stages

#### Candidate

| Property | Definition |
|----------|-----------|
| **Purpose** | A reasoning process is identified as needed. An Attention allocation has selected a stimulus that requires understanding. |
| **Entry** | Attention allocation reaches Evaluate threshold. The system determines that the attended stimulus requires reasoning to be understood. |
| **Exit** | Reasoning is Activated or Retired (if the stimulus is resolved before reasoning begins). |
| **Transitions** | Candidate → Activated (normal progression). Candidate → Retired (stimulus resolved externally). |
| **Failure modes** | Reasoning never activates (Candidate stalls). Reasoning is created but not needed (false selection). |
| **Recovery** | Time-based activation. If Candidate remains in stage beyond threshold, force-activate or retire with Owner notification. |

#### Activated

| Property | Definition |
|----------|-----------|
| **Purpose** | Reasoning begins. The system commits cognitive resources to understanding the selected stimulus. |
| **Entry** | Attention has been allocated, and the reasoning process is initiated. |
| **Exit** | Context Building or Retired. |
| **Transitions** | Activated → Context Building (normal progression). Activated → Retired (reasoning aborted due to resource constraints or Owner cancellation). |
| **Failure modes** | Reasoning activates but cannot proceed (missing prerequisites). Reasoning activates prematurely (context not ready). |
| **Recovery** | If prerequisites are missing, return to Candidate. If context is not ready, defer activation. |

#### Context Building

| Property | Definition |
|----------|-----------|
| **Purpose** | The system gathers the context needed for reasoning: Business Pulse state, current business conditions, temporal context, Owner's current concerns, relevant goals. |
| **Entry** | Reasoning is activated and context collection begins. |
| **Exit** | Evidence Gathering or Retired. |
| **Transitions** | Context Building → Evidence Gathering (context sufficient). Context Building → Retired (context reveals stimulus is no longer relevant). |
| **Failure modes** | Incomplete context. Context stale. Context contaminated (irrelevant or misleading context included). |
| **Recovery** | If context is incomplete, continue with partial context but flag limitations. If context is stale, refresh. If context is contaminated, purify and restart context building. |

#### Evidence Gathering

| Property | Definition |
|----------|-----------|
| **Purpose** | The system collects evidence relevant to the reasoning question. Evidence from the Evidence System, relevant Observations, validated Patterns. |
| **Entry** | Context is established. The system knows what evidence it needs. |
| **Exit** | Knowledge Retrieval or Retired. |
| **Transitions** | Evidence Gathering → Knowledge Retrieval (evidence collected). |
| **Failure modes** | Insufficient evidence. Conflicting evidence. Overwhelming evidence (too much to process). No evidence available. |
| **Recovery** | If insufficient, flag evidence gaps. If conflicting, proceed with conflict analysis. If overwhelming, prioritize evidence by relevance and strength. If none available, proceed with uncertainty. |

#### Knowledge Retrieval

| Property | Definition |
|----------|-----------|
| **Purpose** | The system retrieves relevant Knowledge that constrains or informs reasoning. Validated Knowledge that the system has earned through prior learning. |
| **Entry** | Evidence has been collected. The system knows what Knowledge domains are relevant. |
| **Exit** | Hypothesis Generation or Retired. |
| **Transitions** | Knowledge Retrieval → Hypothesis Generation (Knowledge retrieved). |
| **Failure modes** | Relevant Knowledge not found. Knowledge contradictory with evidence. Knowledge outdated. Knowledge domain unknown. |
| **Recovery** | If Knowledge not found, flag as knowledge gap. If contradictory, prepare contradiction resolution. If outdated, flag for Knowledge refresh. If domain unknown, expand search or flag as exploration opportunity. |

#### Hypothesis Generation

| Property | Definition |
|----------|-----------|
| **Purpose** | The system generates possible explanations for the situation. Hypotheses are candidate understandings that will be tested against evidence and Knowledge. |
| **Entry** | Evidence and Knowledge are available. The system formulates possible explanations. |
| **Exit** | Alternative Generation or Retired. |
| **Transitions** | Hypothesis Generation → Alternative Generation (hypotheses generated). |
| **Failure modes** | Too few hypotheses (tunnel reasoning). Too many hypotheses (analysis paralysis). Hypotheses that do not fit evidence. Hypotheses that contradict established Knowledge without justification. |
| **Recovery** | If too few, force generation of alternatives. If too many, prune by plausibility. If hypotheses do not fit, revisit evidence. If Knowledge contradicted, require explicit contradiction justification. |

#### Alternative Generation

| Property | Definition |
|----------|-----------|
| **Purpose** | The system generates possible courses of action or responses based on the hypotheses. Alternatives are what the system could recommend. |
| **Entry** | Hypotheses are established. The system now considers possible responses. |
| **Exit** | Constraint Evaluation or Retired. |
| **Transitions** | Alternative Generation → Constraint Evaluation (alternatives generated). |
| **Failure modes** | Too few alternatives. Too many alternatives. Alternatives that are not feasible. Alternatives that miss the Owner's preferences. |
| **Recovery** | If too few, force generation. If too many, prune by feasibility. If not feasible, generate realistic alternatives. If Owner preferences unknown, include preference estimation. |

#### Constraint Evaluation

| Property | Definition |
|----------|-----------|
| **Purpose** | The system evaluates each alternative against business constraints: budget, time, resources, regulatory, operational, strategic. |
| **Entry** | Alternatives are generated. The system applies constraints. |
| **Exit** | Tradeoff Evaluation or Retired. |
| **Transitions** | Constraint Evaluation → Tradeoff Evaluation (constraints applied). |
| **Failure modes** | Constraints too restrictive (no alternative survives). Constraints too permissive (everything passes). Constraints missing (important constraint not considered). Constraints incorrectly weighted. |
| **Recovery** | If too restrictive, escalate constraint conflict. If too permissive, tighten constraint thresholds. If missing, identify and add. If incorrectly weighted, recalibrate. |

#### Tradeoff Evaluation

| Property | Definition |
|----------|-----------|
| **Purpose** | The system evaluates the tradeoffs between surviving alternatives: what is gained and lost with each choice, the risks, the opportunities. |
| **Entry** | Constraints have been applied. The system compares remaining alternatives. |
| **Exit** | Consistency Checking or Retired. |
| **Transitions** | Tradeoff Evaluation → Consistency Checking (tradeoffs evaluated). |
| **Failure modes** | Tradeoffs not comparable (apples to oranges). Tradeoffs ignored (alternative selected without tradeoff analysis). Tradeoffs misweighted. |
| **Recovery** | If not comparable, establish common evaluation framework. If ignored, force tradeoff analysis. If misweighted, recalibrate tradeoff criteria. |

#### Consistency Checking

| Property | Definition |
|----------|-----------|
| **Purpose** | The system verifies that the reasoning is internally consistent: conclusions follow from premises, no logical contradictions, no violations of established Knowledge. |
| **Entry** | Tradeoffs are evaluated. The system checks the reasoning chain. |
| **Exit** | Confidence Assessment or Retired (if consistency fails). |
| **Transitions** | Consistency Checking → Confidence Assessment (consistent). Consistency Checking → Hypothesis Generation (inconsistent — revisit hypotheses). |
| **Failure modes** | Internal contradiction detected. Conclusion does not follow from premises. Reasoning violates established Knowledge without justification. Reasoning assumes something not supported. |
| **Recovery** | If contradiction detected, trace the contradiction to its source and resolve. If conclusion does not follow, revisit reasoning steps. If Knowledge violated, justify or abandon. If unsupported assumption, gather support or remove. |

#### Confidence Assessment

| Property | Definition |
|----------|-----------|
| **Purpose** | The system evaluates its confidence in the reasoning outcome. How certain is it that the conclusion is correct? What are the sources of uncertainty? |
| **Entry** | Consistency is verified. The system now assesses how much to trust its own reasoning. |
| **Exit** | Recommendation Building or Retired. |
| **Transitions** | Confidence Assessment → Recommendation Building (confidence sufficient). Confidence Assessment → Hypothesis Generation (confidence too low — revisit). |
| **Failure modes** | Overconfidence (certainty exceeds evidence). Underconfidence (evidence supports but system doubts). Confidence not calibrated. Confidence not communicated. |
| **Recovery** | If overconfident, recalibrate against evidence strength. If underconfident, review evidence that supports conclusion. If not calibrated, adjust confidence model. If not communicated, prepare confidence statement. |

#### Recommendation Building

| Property | Definition |
|----------|-----------|
| **Purpose** | The system constructs the recommendation that will be presented to the Owner or to the Decision system. The recommendation includes the conclusion, the reasoning chain, the confidence level, and the tradeoffs. |
| **Entry** | Confidence is assessed. The system builds the recommendation. |
| **Exit** | Completed or Retired. |
| **Transitions** | Recommendation Building → Completed (recommendation built). Recommendation Building → Hypothesis Generation (recommendation reveals gap — revisit). |
| **Failure modes** | Recommendation not actionable. Recommendation not aligned with Owner preferences. Recommendation missing explanation. Recommendation too complex. |
| **Recovery** | If not actionable, reformulate. If not aligned, incorporate Owner preferences. If missing explanation, build explanation chain. If too complex, simplify. |

#### Completed

| Property | Definition |
|----------|-----------|
| **Purpose** | Reasoning is complete. The recommendation is ready for Decision. |
| **Entry** | Recommendation is built and verified. |
| **Exit** | Archived or Retired. |
| **Transitions** | Completed → Archived (preserved for audit). |
| **Failure modes** | Reasoning completes but recommendation is never used. Reasoning completes but context has changed. Reasoning completes but was unnecessary. |
| **Recovery** | If unused, flag for attention. If context changed, flag for review. If unnecessary, learn to avoid similar reasoning. |

#### Archived

| Property | Definition |
|----------|-----------|
| **Purpose** | The reasoning process is preserved for audit, learning, and future reference. It is no longer active but can be reviewed. |
| **Entry** | Reasoning is Completed and archived. |
| **Exit** | Retired. |
| **Transitions** | Archived → Retired (after retention period or if superseded). |
| **Failure modes** | Archive loss. Archive not discoverable. Archive misleading (reasoning that was later invalidated). |
| **Recovery** | If lost, rebuild from available traces. If not discoverable, index properly. If misleading, mark as superseded. |

#### Retired

| Property | Definition |
|----------|-----------|
| **Purpose** | The reasoning process is permanently concluded. It may be referenced historically but is no longer active or relevant. |
| **Entry** | Archived after retention period, or explicit retirement by Owner. |
| **Exit** | None (terminal stage). |
| **Transitions** | None. |
| **Failure modes** | Premature retirement. Retirement without archiving. |
| **Recovery** | If premature, restore from archive. If unarchived, check if recovery is needed. |

### 2.3 Lifecycle Principles

1. **Forward progression.** Lifecycle moves forward. Stages cannot be skipped (except Candidate → Retired).
2. **Stage re-entry.** Consistency Checking and Confidence Assessment may return to earlier stages if issues are found.
3. **Time bounds.** Each stage has a maximum duration. Exceeding duration triggers escalation.
4. **Traceability.** Every lifecycle transition is recorded.
5. **Owner visibility.** The Owner can view the current lifecycle stage of any reasoning process.

## Part 3 — Reasoning Anatomy

### 3.1 Conceptual Fields

Every reasoning process has the following conceptual fields.

#### Identity

| Property | Definition |
|----------|-----------|
| **What it is** | The unique identity of this reasoning process. Distinguishes it from all other reasoning processes. |
| **Why it exists** | Without identity, reasoning cannot be tracked, interrupted, audited, or referenced by Decision. |
| **Cognitive role** | Provides a fixed reference for the entire reasoning lifecycle and all downstream processes. |

#### Question

| Property | Definition |
|----------|-----------|
| **What it is** | The specific question that the reasoning process is addressing. The cognitive problem to be solved. |
| **Why it exists** | Reasoning must have a clear question. Reasoning without a question is wandering. |
| **Cognitive role** | Defines the scope and goal of the reasoning process. |

#### Intent

| Property | Definition |
|----------|-----------|
| **What it is** | Why this reasoning process exists. The underlying purpose — not the question but the reason the question matters. |
| **Why it exists** | Intent distinguishes between reasoning that serves different goals. Without intent, the system cannot evaluate whether reasoning succeeded. |
| **Cognitive role** | Provides the evaluation criterion for reasoning quality. |

#### Goal

| Property | Definition |
|----------|-----------|
| **What it is** | The desired outcome of this reasoning process. What the system wants to achieve through reasoning. |
| **Why it exists** | Goals constrain reasoning. Different goals lead to different conclusions from the same evidence. |
| **Cognitive role** | Provides the objective function for reasoning evaluation. |

#### Context

| Property | Definition |
|----------|-----------|
| **What it is** | The full cognitive and business context at the time of reasoning: Business Pulse state, business hours, season, current events, Owner's state. |
| **Why it exists** | Context determines what reasoning is appropriate. Context-blind reasoning produces context-blind conclusions. |
| **Cognitive role** | Provides the situational grounding for all reasoning steps. |

#### Inputs

| Property | Definition |
|----------|-----------|
| **What it is** | The complete set of inputs to this reasoning process: Observations, Patterns, Evidence, Memory, Knowledge, Attention allocation. |
| **Why it exists** | Inputs must be recorded for audit, verification, and learning. If the inputs change, reasoning may need to change. |
| **Cognitive role** | Provides the raw material for reasoning. |

#### Attention Sources

| Property | Definition |
|----------|-----------|
| **What it is** | The Attention allocation(s) that triggered this reasoning process. |
| **Why it exists** | Reasoning must trace to its Attention source. Provenance from Attention to Decision must be complete. |
| **Cognitive role** | Enables full cognitive provenance. |

#### Knowledge Sources

| Property | Definition |
|----------|-----------|
| **What it is** | The Knowledge items referenced during reasoning. |
| **Why it exists** | Reasoning relies on Knowledge. Knowledge sources must be traceable for verification and learning. |
| **Cognitive role** | Enables Knowledge-aware reasoning and Knowledge gap detection. |

#### Memory Sources

| Property | Definition |
|----------|-----------|
| **What it is** | The Memory items referenced during reasoning. |
| **Why it exists** | Past experience informs current reasoning. Memory sources must be traceable. |
| **Cognitive role** | Enables experience-informed reasoning. |

#### Evidence Sources

| Property | Definition |
|----------|-----------|
| **What it is** | The Evidence items referenced during reasoning. |
| **Why it exists** | Evidence supports reasoning conclusions. Evidence sources must be traceable for verification. |
| **Cognitive role** | Enables evidence-based reasoning and evidence gap detection. |

#### Patterns

| Property | Definition |
|----------|-----------|
| **What it is** | The Pattern items referenced during reasoning. |
| **Why it exists** | Patterns inform reasoning by showing what has happened before. |
| **Cognitive role** | Enables pattern-informed reasoning. |

#### Constraints

| Property | Definition |
|----------|-----------|
| **What it is** | The business, operational, regulatory, and resource constraints that limit possible conclusions. |
| **Why it exists** | Constraints define the boundaries of valid reasoning. Reasoning that violates constraints is invalid. |
| **Cognitive role** | Provides the boundary conditions for reasoning. |

#### Assumptions

| Property | Definition |
|----------|-----------|
| **What it is** | The assumptions made during reasoning: information taken as true without direct evidence. |
| **Why it exists** | Assumptions must be explicit. Hidden assumptions lead to hidden errors. Explicit assumptions can be verified, challenged, or accepted. |
| **Cognitive role** | Provides the conditional basis for reasoning. |

#### Alternatives

| Property | Definition |
|----------|-----------|
| **What it is** | The alternative explanations, conclusions, or courses of action considered during reasoning. |
| **Why it exists** | Reasoning without alternatives is not reasoning — it is confirmation. Alternatives ensure the system has considered multiple possibilities. |
| **Cognitive role** | Enables comparative evaluation. |

#### Tradeoffs

| Property | Definition |
|----------|-----------|
| **What it is** | The tradeoffs evaluated between alternatives: what is gained and lost with each choice. |
| **Why it exists** | Every business decision involves tradeoffs. Reasoning must make tradeoffs explicit. |

## Part 4 — Reasoning Types

### 4.1 Type Framework

Every reasoning process belongs to a type. Type determines the reasoning approach, the inputs required, the constraints applied, and the output format.

#### Diagnostic Reasoning

| Property | Definition |
|----------|-----------|
| **Purpose** | Determine why something happened. Identify root causes from symptoms. |
| **Question** | "Why did X happen?" |
| **Approach** | Trace from symptom back to cause. Eliminate competing explanations. |
| **Typical inputs** | Observations (symptoms), Patterns (known symptom-cause mappings), Knowledge (causal relationships), Memory (similar past events). |
| **Typical outputs** | Root cause identification, contributing factors, confidence in diagnosis. |
| **Business examples** | Why did sales decline? Why is waste increasing? Why are customers complaining? |

#### Strategic Reasoning

| Property | Definition |
|----------|-----------|
| **Purpose** | Determine long-term direction. Evaluate strategic options. |
| **Question** | "What should we do over the next N months/years?" |
| **Approach** | Evaluate alternatives against long-term goals, market conditions, competitive position, resources. |
| **Typical inputs** | Market Knowledge, competitive analysis, business goals, resource constraints, strategic patterns. |
| **Typical outputs** | Strategic recommendation, timeline options, risk assessment. |
| **Business examples** | Should we expand to a second location? Should we change our menu concept? |

#### Operational Reasoning

| Property | Definition |
|----------|-----------|
| **Purpose** | Determine day-to-day operational decisions. Optimize current operations. |
| **Question** | "What should we do right now / today / this week?" |
| **Approach** | Evaluate operational alternatives against current conditions and immediate constraints. |
| **Typical inputs** | Current operational data, staffing levels, inventory status, service metrics, Knowledge of operational patterns. |
| **Typical outputs** | Operational recommendation, immediate action plan. |
| **Business examples** | How should we adjust staffing for tonight's reservation level? What should we do about the broken refrigerator? |

#### Financial Reasoning

| Property | Definition |
|----------|-----------|
| **Purpose** | Analyze financial performance, evaluate financial decisions. |
| **Question** | "What is the financial impact of X?" |
| **Approach** | Apply financial Knowledge to business data. Evaluate costs, revenues, margins, ROI. |
| **Typical inputs** | Financial data, cost Knowledge, pricing Knowledge, historical financial patterns. |
| **Typical outputs** | Financial analysis, cost-benefit assessment, financial risk evaluation. |
| **Business examples** | Should we raise prices? Is this supplier cost-effective? What is the ROI of this equipment purchase? |

#### Risk Reasoning

| Property | Definition |
|----------|-----------|
| **Purpose** | Identify, evaluate, and prioritize risks. |
| **Question** | "What could go wrong and how bad would it be?" |
| **Approach** | Identify risk factors, evaluate likelihood and impact, prioritize, recommend mitigation. |
| **Typical inputs** | Risk Knowledge, historical risk patterns, current conditions, operational data. |
| **Typical outputs** | Risk inventory, risk ranking, mitigation recommendations. |
| **Business examples** | What are the risks of this promotion? What are the food safety risks? |

#### Customer Reasoning

| Property | Definition |
|----------|-----------|
| **Purpose** | Understand customer behavior, preferences, satisfaction. |
| **Question** | "What do our customers want and how do they feel?" |
| **Approach** | Analyze customer data against customer Knowledge and behavioral patterns. |
| **Typical inputs** | Customer feedback, behavior data, satisfaction scores, demographic patterns, customer Knowledge. |
| **Typical outputs** | Customer insight, satisfaction analysis, recommendation for customer experience improvement. |
| **Business examples** | Why are customers leaving negative reviews? What menu items do customers prefer? |

#### Inventory Reasoning

| Property | Definition |
|----------|-----------|
| **Purpose** | Optimize inventory levels, reduce waste, ensure availability. |
| **Question** | "How much should we order and when?" |
| **Approach** | Analyze usage patterns, lead times, seasonal variations, storage constraints, cost of waste vs. cost of stockout. |
| **Typical inputs** | Inventory levels, usage patterns, supplier lead times, waste data, Knowledge of inventory dynamics. |
| **Typical outputs** | Order recommendations, waste reduction recommendations, stockout risk assessment. |
| **Business examples** | How much should we order for next week? Why is waste increasing? |

#### Learning Reasoning

| Property | Definition |
|----------|-----------|
| **Purpose** | Determine what the system can learn from an experience. Extract Knowledge from outcomes. |
| **Question** | "What can we learn from what happened?" |
| **Approach** | Analyze the gap between expectation and outcome. Determine if the gap reveals Knowledge gaps, pattern changes, or new understanding. |
| **Typical inputs** | Past reasoning processes, outcomes, prediction deviations, unexpected results. |
| **Typical outputs** | Learning recommendations, Knowledge gap identification, pattern update recommendations. |
| **Business examples** | What did we learn from the failed promotion? Why was our prediction wrong? |

#### Predictive Reasoning

| Property | Definition |
|----------|-----------|
| **Purpose** | Forecast future states based on patterns, Knowledge, and current conditions. |
| **Question** | "What will happen if current trends continue?" |
| **Approach** | Apply Knowledge of causal relationships and patterns to current conditions. Evaluate confidence in predictions. |
| **Typical inputs** | Current conditions, historical patterns, Knowledge of causal relationships, trend data. |
| **Typical outputs** | Prediction with confidence interval, prediction sensitivity analysis. |
| **Business examples** | What will sales be next month? When will we run out of inventory? |

#### Planning Reasoning

| Property | Definition |
|----------|-----------|
| **Purpose** | Determine the sequence of actions to achieve a goal. |
| **Question** | "What steps do we need to take to achieve X?" |
| **Approach** | Decompose goal into sub-goals. Determine dependencies, order, resources, timeline. |
| **Typical inputs** | Goal, constraints, available resources, Knowledge of effective approaches, patterns of success/failure. |
| **Typical outputs** | Action sequence, timeline, resource requirements, risk assessment. |
| **Business examples** | How should we prepare for the health inspection? What is the plan for the new menu launch? |

#### Meta Reasoning

| Property | Definition |
|----------|-----------|
| **Purpose** | Reason about reasoning itself. Evaluate the system's own cognitive process. |
| **Question** | "Is our reasoning process working correctly?" |
| **Approach** | Analyze reasoning patterns, quality assessments, failure modes, lifecycle efficiency. |
| **Typical inputs** | Reasoning quality profiles, lifecycle data, failure mode records, comparative analysis. |
| **Typical outputs** | Reasoning process improvement recommendations. |
| **Business examples** | Why did we miss that critical issue? Is our reasoning taking too long? |

#### Reflective Reasoning

| Property | Definition |
|----------|-----------|
| **Purpose** | Review past reasoning to improve future reasoning. |
| **Question** | "What could we have done better?" |
| **Approach** | Compare reasoning conclusions with actual outcomes. Identify reasoning errors, gaps, overconfidence, underconfidence. |
| **Typical inputs** | Past reasoning processes, actual outcomes, outcome analysis. |
| **Typical outputs** | Reflective assessment, improvement recommendations, Knowledge updates. |
| **Business examples** | Was our reasoning about the supplier replacement correct? What did we miss in the equipment failure analysis? |

#### Ethical Reasoning

| Property | Definition |
|----------|-----------|
| **Purpose** | Evaluate the ethical implications of decisions and recommendations. |
| **Question** | "Is this the right thing to do?" |
| **Approach** | Apply ethical principles from the Constitution. Evaluate impacts on customers, staff, community, environment. |
| **Typical inputs** | Constitution, ethical principles, stakeholder impacts, regulatory Knowledge. |
| **Typical outputs** | Ethical assessment, ethical risk identification, recommendation for ethical action. |
| **Business examples** | Is it ethical to raise prices during a supply shortage? Should we serve customers who have had too much to drink? |

#### Composite Reasoning

| Property | Definition |
|----------|-----------|
| **Purpose** | Integrate multiple reasoning types to address complex situations. |
| **Question** | "What is happening and what should we do?" (Multiple sub-questions). |
| **Approach** | Decompose into sub-reasoning processes of different types. Synthesize results. |
| **Typical inputs** | All inputs from constituent reasoning types. |
| **Typical outputs** | Integrated recommendation combining insights from multiple perspectives. |
| **Business examples** | Comprehensive restaurant health decline analysis (diagnostic + operational + risk + customer). Expansion decision (strategic + financial + risk + planning). |

### 4.2 Type Selection Principles

1. **Type is determined by question.** The question determines the reasoning type.
2. **Some situations require multiple types.** Complex situations use Composite Reasoning.
3. **Type determines lifecycle behavior.** Different types may have different stage durations.
4. **Type determines quality weighting.** Some quality dimensions matter more for certain types (e.g., Creativity matters more for Strategic Reasoning than Diagnostic Reasoning).

## Part 5 — Reasoning Quality

### 5.1 Quality Dimensions

Every reasoning process is assessed across fourteen quality dimensions.

#### Correctness

| Property | Definition |
|----------|-----------|
| **Meaning** | Whether the reasoning conclusion corresponds to reality when validated against outcomes. |
| **Assessment** | Comparison of reasoning conclusion with actual outcome. |
| **Failure detection** | Conclusion does not match outcome. The system recommended X but the correct answer was Y. |
| **Recovery** | (1) Identify the reasoning step where error occurred. (2) Determine root cause (wrong evidence? wrong Knowledge? wrong logic?). (3) Correct the specific failure. (4) If systemic, adjust reasoning process. |

#### Consistency

| Property | Definition |
|----------|-----------|
| **Meaning** | Whether the reasoning is internally consistent: conclusions follow from premises, no contradictions, assumptions align with conclusions. |
| **Assessment** | Internal audit of the reasoning chain. Cross-check each step against previous steps. |
| **Failure detection** | Contradiction detected: Step A says X, Step B says not-X. Conclusion does not follow from premises. Assumptions contradict conclusions. |
| **Recovery** | (1) Identify inconsistency location. (2) Resolve by determining which premise is correct. (3) If premises are both valid, the contradiction indicates missing Knowledge. (4) Flag for Knowledge gap resolution. |

#### Completeness

| Property | Definition |
|----------|-----------|
| **Meaning** | Whether the reasoning considered all relevant factors: all alternatives, all evidence, all constraints, all risks. |
| **Assessment** | Comparison of considered factors with an ideal complete set. |
| **Failure detection** | After-action review reveals factors that were not considered. Owner says "did you consider X?" |
| **Recovery** | (1) Identify missing factor. (2) Determine why it was missed. (3) Re-run reasoning with missing factor. (4) If systemic, improve coverage. |

#### Depth

| Property | Definition |
|----------|-----------|
| **Meaning** | Whether the reasoning goes beyond surface-level analysis to understand root causes, second-order effects, and systemic implications. |
| **Assessment** | Number of causal layers explored. Consideration of indirect effects. |
| **Failure detection** | Recommendation addresses symptoms but not causes. Second-order effects missed. Systemic implications not considered. |
| **Recovery** | (1) Identify depth gap. (2) Probe deeper causal layers. (3) Consider second-order effects. (4) Expand reasoning scope. |

#### Breadth

| Property | Definition |
|----------|-----------|
| **Meaning** | Whether the reasoning considered a wide enough range of alternatives, perspectives, and factors. |
| **Assessment** | Number of alternatives generated. Diversity of perspectives considered. Range of factors evaluated. |
| **Failure detection** | Too few alternatives generated. Same perspective applied to all analyses. Factors from only one domain considered. |
| **Recovery** | (1) Force generate additional alternatives. (2) Seek alternative perspectives. (3) Expand factor consideration. (4) If limited breadth is intentional, document reasoning. |

#### Creativity

| Property | Definition |
|----------|-----------|
| **Meaning** | Whether the reasoning generated novel approaches or insights, not just standard solutions. |
| **Assessment** | Novelty of alternatives compared to standard approaches. |
| **Failure detection** | All alternatives are standard. No novel insight generated. Recommendation is what anyone would have said. |
| **Recovery** | (1) Generate non-standard alternatives. (2) Challenge assumptions. (3) Consider unconventional approaches. (4) If creativity is not required (routine reasoning), document intent. |

#### Novelty

| Property | Definition |
|----------|-----------|
| **Meaning** | Whether the reasoning produces new understanding, not just confirmation of what the system already knows. |
| **Assessment** | Comparison of reasoning output with existing Knowledge. Does it add anything new? |
| **Failure detection** | Reasoning confirms existing Knowledge without adding new insight. Reasoning repeats what was already known. |
| **Recovery** | (1) Identify what new understanding was expected. (2) Determine if the situation genuinely required novel reasoning or was routine. (3) If routine, mark as confirmation reasoning. (4) If novel, investigate why no new understanding emerged. |

#### Explainability

| Property | Definition |
|----------|-----------|
| **Meaning** | Whether the reasoning chain can be communicated to the Owner in a clear, understandable way. |
| **Assessment** | Can the system produce a natural-language explanation of how it reached its conclusion? |
| **Failure detection** | The system cannot explain its reasoning. The reasoning chain has gaps that cannot be articulated. |
| **Recovery** | (1) Reconstruct reasoning chain. (2) Identify gaps. (3) Fill gaps or flag as uncertainty. (4) If systemic, improve reasoning documentation. |

#### Business Alignment

| Property | Definition |
|----------|-----------|
| **Meaning** | Whether the reasoning serves business goals and aligns with the Constitution and Product Principles. |
| **Assessment** | Traceability from reasoning to business goals. Consistency with Constitution. |
| **Failure detection** | Reasoning conclusion conflicts with business goals. Recommendation would harm business outcomes. |
| **Recovery** | (1) Identify misalignment. (2) Re-evaluate against business goals. (3) Adjust reasoning to align. (4) If alignment impossible due to constraint conflict, escalate. |

#### Robustness

| Property | Definition |
|----------|-----------|
| **Meaning** | Whether the reasoning conclusion holds under different assumptions or changed conditions. |
| **Assessment** | Sensitivity analysis: change assumptions, inputs, or context and observe whether conclusion changes. |
| **Failure detection** | Small change in assumption leads to dramatically different conclusion (brittle reasoning). Conclusion only valid under narrow conditions. |
| **Recovery** | (1) Identify brittle points. (2) Strengthen weak assumptions. (3) Expand conditions under which conclusion holds. (4) If conclusion is inherently conditional, document conditions explicitly. |

#### Efficiency

| Property | Definition |
|----------|-----------|
| **Meaning** | Whether the reasoning produced value proportional to the cognitive resources consumed. |
| **Assessment** | Cognitive cost vs. value of recommendation. |
| **Failure detection** | Reasoning took too long or consumed too many resources for the value produced. Over-analysis of simple situations. Under-analysis of complex situations. |
| **Recovery** | (1) Assess efficiency failure pattern. (2) Adjust resource allocation for similar future reasoning. (3) If over-analysis, implement complexity-based resource allocation. |

#### Owner Satisfaction

| Property | Definition |
|----------|-----------|
| **Meaning** | Whether the Owner finds the reasoning helpful, clear, and trustworthy. |
| **Assessment** | Owner feedback. Owner's acceptance rate of recommendations. |
| **Failure detection** | Owner rejects recommendations. Owner asks clarifying questions that reasoning should have addressed. Owner expresses frustration with reasoning quality. |
| **Recovery** | (1) Gather specific feedback. (2) Identify specific failure. (3) Correct. (4) If pattern of dissatisfaction, systemic review. |

#### Confidence Calibration

| Property | Definition |
|----------|-----------|
| **Meaning** | Whether the system's confidence matches actual correctness. High confidence when correct, low confidence when wrong. |
| **Assessment** | Correlation between confidence and correctness across many reasoning processes. |
| **Failure detection** | High confidence with wrong conclusions. Low confidence with correct conclusions. Confidence does not vary with evidence strength. |
| **Recovery** | (1) Identify calibration gap. (2) Adjust confidence estimation. (3) If systemic, recalibrate confidence model. |

#### Timeliness

| Property | Definition |
|----------|-----------|
| **Meaning** | Whether the reasoning completes within the required time window. |
| **Assessment** | Comparison of completion time with time window defined by Attention allocation. |
| **Failure detection** | Reasoning completes after deadline. Recommendation arrives too late to be useful. |
| **Recovery** | (1) Assess timeliness failure. (2) If recurring, adjust reasoning complexity or resource allocation. (3) If reasoning could not complete in time, implement time-aware reasoning. |

## Part 7 — Reasoning Pipeline

### 7.1 Pipeline Overview

The reasoning pipeline is the cognitive flow through which attention is transformed into understanding and recommendation. This is not an implementation. It is a description of the cognitive sequence.

### 7.2 Pipeline Stages

#### Stage 1: Receive Attention

The reasoning process receives an Attention allocation. The Attention allocation carries:
- The stimulus that earned Attention
- The context at time of allocation
- The priority and urgency
- The source (Perception, Owner, Business Pulse, etc.)

**Cognitive check:** Is the stimulus appropriate for reasoning? Some stimuli require immediate action, not reasoning. If the stimulus is below reasoning threshold, return to Attention for action routing.

#### Stage 2: Collect Context

The reasoning process gathers the context needed:
- Current Business Pulse state
- Business hours indicator
- Current season and date context
- Recent significant events
- Owner's current focus or concerns
- Active goals and their progress

**Cognitive check:** Is context sufficient? If critical context elements are missing, flag context gaps and proceed with partial context or defer.

#### Stage 3: Retrieve Knowledge

The reasoning process retrieves Knowledge relevant to the question:
- Causal Knowledge (X causes Y)
- Operational Knowledge (how processes work)
- Strategic Knowledge (what works long-term)
- Financial Knowledge (cost, revenue, margin relationships)
- Regulatory Knowledge (compliance requirements)
- Domain-specific Knowledge (inventory, staffing, customer behavior, etc.)

**Cognitive check:** Is Knowledge sufficient for reasoning? If critical Knowledge domains are missing, flag Knowledge gaps. If Knowledge is contradictory with current context, flag contradiction.

#### Stage 4: Retrieve Memory

The reasoning process retrieves relevant Memory:
- Similar past situations
- Previous reasoning on similar questions
- Past outcomes of similar decisions
- Lessons learned from past experiences

**Cognitive check:** Is Memory relevant? Past situations are never exactly identical. The system must assess similarity and applicability, not blindly apply past conclusions.

#### Stage 5: Retrieve Evidence

The reasoning process retrieves Evidence relevant to the question:
- Current Observations
- Validated Patterns
- Supporting Evidence from the Evidence System
- Data points relevant to the reasoning question

**Cognitive check:** Is Evidence sufficient? If evidence is insufficient, flag evidence gaps. If evidence is contradictory, flag for contradiction resolution.

#### Stage 6: Formulate Question

The reasoning process formulates the precise question to be answered:
- What is the exact question?
- What type of reasoning is needed?
- What is the scope of the question?
- What would constitute a good answer?

**Cognitive check:** Is the question well-formed? A poorly formed question guarantees poor reasoning.

#### Stage 7: Generate Hypotheses

The reasoning process generates possible explanations or answers:
- Hypothesis A: the most likely explanation
- Hypothesis B: an alternative explanation
- Hypothesis C: a less likely but possible explanation
- At minimum, the "null hypothesis" — the situation is normal, no action needed

**Cognitive check:** Are hypotheses covering the possibility space? If all hypotheses cluster around one explanation, force alternative generation.

#### Stage 8: Generate Alternatives

The reasoning process generates possible courses of action:
- Alternative 1: the standard approach
- Alternative 2: a conservative approach
- Alternative 3: an aggressive approach
- Alternative 4: do nothing (baseline)

**Cognitive check:** Are alternatives genuinely different? If alternatives are the same approach with minor variations, force genuinely different alternatives.

#### Stage 9: Evaluate Constraints

The reasoning process evaluates each hypothesis and alternative against constraints:
- Business constraints (budget, resources, time)
- Operational constraints (staff capacity, equipment, space)
- Regulatory constraints (health code, labor law, licensing)
- Strategic constraints (brand positioning, long-term goals)
- Knowledge constraints (what the system knows to be true)

**Cognitive check:** Do any constraints eliminate all alternatives? If so, the constraint conflict must be escalated.

#### Stage 10: Evaluate Tradeoffs

The reasoning process evaluates tradeoffs between surviving alternatives:
- What is gained with each alternative?
- What is lost with each alternative?
- What risks does each alternative carry?
- What opportunities does each alternative open or close?

**Cognitive check:** Are tradeoffs comparable? If tradeoffs are in fundamentally different domains (e.g., cost vs. reputation), establish a common evaluation framework.

#### Stage 11: Check Consistency

The reasoning process verifies internal consistency:
- Do conclusions follow from premises?
- Are there any logical contradictions?
- Are all assumptions explicit and justified?
- Does the reasoning violate any established Knowledge?
- Is the reasoning chain complete?

**Cognitive check:** If consistency fails, trace the inconsistency to its source and resolve. Do not proceed with inconsistent reasoning.

#### Stage 12: Assess Confidence

The reasoning process assesses confidence:
- How strong is the evidence?
- How reliable is the Knowledge?
- How complete is the context?
- How robust is the conclusion to assumption changes?
- What is the uncertainty range?

**Cognitive check:** Is confidence calibrated to evidence strength? If confidence exceeds what evidence supports, flag overconfidence. If confidence is lower than evidence warrants, flag underconfidence.

#### Stage 13: Build Recommendation

The reasoning process constructs the recommendation:
- The recommended alternative
- The reasoning chain (how we got here)
- The confidence level and uncertainty sources
- The tradeoffs and risks of the recommended alternative
- Alternative options with their tradeoffs
- The expected outcome of the recommendation

**Cognitive check:** Is the recommendation actionable? Can the Owner understand it? Does it include enough information for decision?

#### Stage 14: Prepare Decision Support

The reasoning process prepares the output for the Decision system:
- Structured recommendation
- Confidence assessment
- Risk evaluation
- Expected outcomes
- Alternative options
- Owner alignment assessment
- Time sensitivity
- Required action type

**Cognitive check:** Is the decision support complete? Can the Decision system act on it?

### 7.3 Pipeline Principles

1. **Pipeline is not strictly sequential.** Stages may repeat, loop back, or skip if appropriate.
2. **Pipeline is not implementation.** It is a cognitive description.
3. **Pipeline completion is not mandatory for all stimuli.** Some stimuli require immediate action. The pipeline applies when reasoning is needed.
4. **Pipeline quality is proportional to reasoning quality.** Rushing the pipeline reduces reasoning quality.

## Part 8 — Reasoning Policies

### 8.1 Policy Categories

#### Uncertainty Policy

| Property | Definition |
|----------|-----------|
| **What it governs** | How reasoning handles uncertainty in inputs, Knowledge, and context. |
| **Rules** | (1) Uncertainty must be explicitly identified and categorized (epistemic, aleatory, model, measurement). (2) Uncertainty must be reflected in confidence assessment. (3) Reasoning cannot assume certainty where uncertainty exists. (4) When uncertainty exceeds threshold, flag for Owner. |
| **Rationale** | Ensures honest reasoning that acknowledges its limitations. |

#### Contradiction Policy

| Property | Definition |
|----------|-----------|
| **What it governs** | How reasoning handles contradictions in evidence, Knowledge, or Memory. |
| **Rules** | (1) Contradictions must be explicitly identified, not silently resolved. (2) The system must attempt to resolve contradictions by evaluating evidence strength. (3) If contradictions cannot be resolved, both sides must be presented with their supporting evidence. (4) Unresolved contradictions must be flagged for Learning (Knowledge gap). |
| **Rationale** | Prevents silent suppression of contradictions, which leads to false certainty. |

#### Missing Evidence Policy

| Property | Definition |
|----------|-----------|
| **What it governs** | How reasoning proceeds when evidence is insufficient or missing. |
| **Rules** | (1) Missing evidence must be explicitly identified. (2) Reasoning can proceed with available evidence but must flag gaps. (3) Confidence must be reduced proportional to evidence gaps. (4) If critical evidence is missing, reasoning must recommend evidence collection before decision. |
| **Rationale** | Prevents false confidence based on incomplete evidence. |

#### Conflicting Knowledge Policy

| Property | Definition |
|----------|-----------|
| **What it governs** | How reasoning handles Knowledge that conflicts with other Knowledge or with evidence. |
| **Rules** | (1) Knowledge conflicts must be identified and resolved before reasoning completes. (2) Resolution follows Knowledge System rules: validated Knowledge takes precedence over deprecated Knowledge. (3) If Knowledge conflict involves two validated Knowledge items, the conflict indicates a Knowledge gap and must trigger Learning. (4) Reasoning cannot proceed while a validated Knowledge conflict exists. |
| **Rationale** | Ensures reasoning operates on the best available Knowledge. |

#### Owner Preferences Policy

| Property | Definition |
|----------|-----------|
| **What it governs** | How reasoning incorporates the Owner's stated and observed preferences. |
| **Rules** | (1) Owner preferences are the highest-weight factor in alternative evaluation (after constraints). (2) Observed Owner behavior informs preference estimation when explicit statements are unavailable. (3) If a recommended alternative conflicts with Owner preferences, the conflict must be flagged. (4) Owner preferences can override standard reasoning weights. |
| **Rationale** | The Owner is sovereign. Reasoning must serve the Owner, not itself. |

#### Business Constraints Policy

| Property | Definition |
|----------|-----------|
| **What it governs** | How reasoning respects business constraints. |
| **Rules** | (1) Business constraints are mandatory, not advisory. (2) If no alternative survives constraint evaluation, the constraints must be escalated (not relaxed). (3) Constraint relaxation requires Owner approval. (4) Constraint changes must be documented. |
| **Rationale** | Constraints exist for a reason. Reasoning does not override them. |

#### Incomplete Context Policy

| Property | Definition |
|----------|-----------|
| **What it governs** | How reasoning proceeds when context is incomplete. |
| **Rules** | (1) Incomplete context must be explicitly identified. (2) Reasoning can proceed with partial context but must flag gaps. (3) Confidence must reflect context completeness. (4) If critical context elements are missing, reasoning must defer or recommend context collection. |
| **Rationale** | Context-blind reasoning produces context-blind conclusions. |

#### Time Pressure Policy

| Property | Definition |
|----------|-----------|
| **What it governs** | How reasoning adapts to time constraints. |
| **Rules** | (1) Time pressure affects reasoning depth, not reasoning quality. (2) Under time pressure, reasoning may use fewer iterations, fewer alternatives, higher confidence thresholds. (3) Time-pressured reasoning must flag its reduced scope. (4) If time pressure prevents adequate reasoning, the system must recommend immediate action with reduced confidence. |
| **Rationale** | Time pressure is real. Reasoning must adapt without pretending it had more time. |

#### Emergency Policy

| Property | Definition |
|----------|-----------|
| **What it governs** | How reasoning behaves during emergency conditions. |
| **Rules** | (1) Emergency reasoning bypasses standard depth requirements. (2) Emergency reasoning must produce the fastest actionable recommendation. (3) Post-emergency, full reasoning must be conducted as reflection. (4) Emergency reasoning is flagged as emergency-mode for quality assessment. |
| **Rationale** | During emergencies, speed is prioritized over depth. But the system must learn from emergency reasoning afterward. |

#### Risk Policy

| Property | Definition |
|----------|-----------|
| **What it governs** | How reasoning evaluates and communicates risk. |
| **Rules** | (1) Every alternative must have a risk assessment. (2) Risk assessment includes likelihood and impact. (3) High-risk alternatives must have explicit risk mitigation recommendations. (4) If all alternatives carry high risk, the system must flag the situation as high-risk. (5) Risk tolerance is set by the Owner. |
| **Rationale** | Risk is inherent in business decisions. Reasoning must make risk explicit. |

#### Recommendation Policy

| Property | Definition |
|----------|-----------|
| **What it governs** | How recommendations are structured and presented. |
| **Rules** | (1) Every recommendation must include a reasoning chain. (2) Every recommendation must include confidence and uncertainty. (3) Every recommendation must include alternatives considered. (4) Every recommendation must include tradeoffs. (5) Every recommendation must include expected outcomes. (6) Recommendations are advisory, not directive. The Owner decides. |
| **Rationale** | Recommendations are decision support, not decisions. |

### 8.2 Policy Conflict Resolution

When policies conflict, resolution follows this hierarchy:

1. **Emergency Policy** — highest priority
2. **Owner Preferences Policy** — Owner sovereignty
3. **Business Constraints Policy** — constraints are mandatory
4. **Contradiction Policy** — must resolve contradictions
5. **Risk Policy** — must evaluate risk
6. **Remaining policies** — resolved by context

### 8.3 Policy Principles

1. **Policies guide, not dictate.** Reasoning is not mechanical rule-following.
2. **Policy violations must be documented.** If reasoning cannot follow a policy, the exception must be recorded.
3. **Policies evolve.** As the system learns, policies may be refined.
4. **Policies are visible to the Owner.** The Owner can review and adjust policies.

## Part 9 — Reasoning Invariants

Every invariant is a rule that must hold true for the Reasoning System to be correct. Invariants are grouped by category.

| ID | Invariant |
|----|-----------|
| INV-RSN-001 | Every reasoning process has exactly one unique identity at any time. |
| INV-RSN-002 | Reasoning identity is immutable once assigned. |
| INV-RSN-003 | Reasoning identity must be distinct from all other cognitive entity identities (Observation, Memory, Knowledge, Pattern, Evidence, Attention, Decision, Planning, Prediction, Learning). |
| INV-RSN-004 | A reasoning process's identity persists across its entire lifecycle (Candidate through Retired). |
| INV-RSN-005 | An identity cannot be reused for a different reasoning process, even after the original process is Retired. |

### Category 2: Lifecycle Invariants (INV-RSN-006 to INV-RSN-025)

| ID | Invariant |
|----|-----------|
| INV-RSN-006 | Every reasoning process exists in exactly one lifecycle stage at any time. |
| INV-RSN-007 | Lifecycle transitions are forward (no regression except Consistency Checking and Confidence Assessment may return to Hypothesis Generation). |
| INV-RSN-008 | Candidate is always the initial stage. Every reasoning process begins as Candidate. |
| INV-RSN-009 | Only Candidate processes can be Activated. |
| INV-RSN-010 | Activation requires at least one Attention allocation as source. |
| INV-RSN-011 | Context Building must complete before Evidence Gathering begins (context must be established before evidence can be evaluated). |

| INV-RSN-013 | Knowledge Retrieval must complete before Hypothesis Generation (Knowledge constrains possible hypotheses). |
| INV-RSN-014 | Hypothesis Generation must complete before Alternative Generation (alternatives are responses to hypotheses). |
| INV-RSN-015 | Alternative Generation must complete before Constraint Evaluation (constraints apply to alternatives). |
| INV-RSN-016 | Constraint Evaluation must complete before Tradeoff Evaluation (constraints determine which alternatives survive). |
| INV-RSN-017 | Tradeoff Evaluation must complete before Consistency Checking (tradeoffs are part of the reasoning chain). |
| INV-RSN-018 | Consistency Checking must complete before Confidence Assessment (consistency is a prerequisite for confidence). |
| INV-RSN-019 | Confidence Assessment must complete before Recommendation Building (confidence informs recommendation presentation). |

| INV-RSN-021 | Only Completed processes can be Archived. |

## Part 10 — Reasoning Failure Modes

### 10.1 Failure Mode Classification

Every failure mode is classified by its cognitive origin and follows a consistent structure: Definition, Symptoms, Root Causes, Detection, Severity, Prevention, Recovery. All failure modes are organized into ten categories.

### 10.2 Cognitive Failures

Failures that originate in the reasoning process itself — errors in how reasoning is performed.

#### 10.2.1 Circular Reasoning

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process uses its conclusion as a premise. A proves B because of C, but C assumes B. The reasoning chain contains a logical cycle. |
| **Symptoms** | Reasoning chain loops back on itself. No new information is introduced. The system appears to go in circles. Output does not add understanding. |
| **Root causes** | (1) Missing external inputs. (2) Insufficient Knowledge forcing self-reference. (3) Inadequate consistency checking. (4) Reasoning process trapped in a cycle without detection. |
| **Detection** | The reasoning graph contains a cycle. A premise references a conclusion. Consistency check reveals self-reference. |
| **Severity** | Critical — circular reasoning is not reasoning. It is self-deception dressed as logic. |
| **Prevention** | Implement cycle detection in consistency checking. Require at least one external input per reasoning step. Enforce acyclic reasoning chain structure. |
| **Recovery** | Identify the circular dependency. Break the cycle by requiring a new external input. If no external input is available, flag the reasoning as incomplete and escalate. |

#### 10.2.2 Confirmation Bias

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process favors evidence and Knowledge that confirms its preferred conclusion while systematically ignoring or discounting contradicting evidence. |
| **Symptoms** | All evidence supports the conclusion. Contradicting evidence is absent or dismissed with weak justification. Alternatives are not genuinely considered. Evidence retrieval is one-sided. |
| **Root causes** | (1) Hypothesis generated before evidence gathered. (2) Unequal weighting of confirming versus disconfirming evidence. (3) Tunnel vision in evidence retrieval strategy. (4) Unconscious preference for the initial hypothesis. |
| **Detection** | Evidence distribution is skewed toward confirming evidence (greater than 80% supporting). Disconfirming evidence exists but is not referenced. |
| **Severity** | High — leads to systematically wrong conclusions across multiple reasoning processes. |
| **Prevention** | Always retrieve disconfirming evidence. Implement balanced evidence retrieval policy. Force alternative generation before evaluation. Mandate counter-hypothesis generation. |
| **Recovery** | Force retrieval of disconfirming evidence. Re-evaluate evidence weighting. Generate counter-hypotheses. Re-run the reasoning with balanced inputs. |

#### 10.2.3 Tunnel Reasoning

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process narrows its focus too early, considering only one explanation or approach and systematically ignoring viable alternatives. |
| **Symptoms** | Single hypothesis from the start. No genuine alternatives generated. Conclusion appears predetermined from early stages. Analysis remains surface-level. |
| **Root causes** | (1) Premature hypothesis commitment. (2) Overconfidence in initial assessment. (3) Time pressure causing early narrowing. (4) Cognitive inertia — difficulty shifting from initial frame. |
| **Detection** | Alternative count is below minimum threshold. All alternatives are minor variations of the same core approach. |
| **Severity** | High — leads to missed solutions, overlooked risks, and suboptimal recommendations. |
| **Prevention** | Enforce minimum alternative count per reasoning type. Implement mandatory alternative generation phase. Require at least one alternative that challenges the initial hypothesis. |
| **Recovery** | Force generation of additional alternatives that differ fundamentally from the initial approach. Challenge the initial hypothesis with counter-evidence. |

#### 10.2.4 Reasoning Collapse

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process fails to produce any conclusion or recommendation. The system cannot complete the reasoning lifecycle. All cognitive effort produces no output. |
| **Symptoms** | Reasoning stalls at a stage indefinitely. Repeated stage re-entries without progress. No recommendation produced. Process exceeds maximum duration consistently. |
| **Root causes** | (1) Insufficient inputs to form any hypothesis. (2) Unresolvable contradictions blocking all paths. (3) No alternative survives constraint evaluation. (4) Confidence never reaches threshold regardless of iteration. |
| **Detection** | Lifecycle duration exceeds maximum with no completion in sight. Stage re-entry count exceeds limit. |
| **Severity** | Critical — the system cannot produce understanding when it is most needed. |
| **Prevention** | Implement reasoning collapse detection. Set maximum reasoning duration per reasoning type. Establish escalation path for blocked reasoning. Provide fallback recommendation mode. |
| **Recovery** | Escalate immediately. Identify the blocking issue explicitly. If no alternative survives constraints, escalate the constraint conflict to the Owner. If contradictions cannot be resolved, present both sides with documented uncertainty. |

#### 10.2.5 Analysis Paralysis

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process continues analyzing indefinitely, generating more alternatives, more evidence, more tradeoffs, without converging on a conclusion. Analysis becomes an end in itself. |
| **Symptoms** | Increasing number of alternatives without convergence. Diminishing returns per additional analysis cycle. Never reaching recommendation stage. Iterations increase without bound. |
| **Root causes** | (1) Perfectionism — seeking the perfect answer. (2) Fear of wrong conclusion. (3) Insufficient stopping criteria. (4) No decision deadline or time pressure. |
| **Detection** | Analysis-to-conclusion ratio is excessive. Alternatives increase without convergence. Stage re-entry count exceeds threshold for the reasoning type. |
| **Severity** | Medium — delays decision but may produce better analysis in theory. Cost of delay may exceed value of additional analysis. |
| **Prevention** | Set maximum alternatives per reasoning type. Implement diminishing returns detection. Enforce decision deadlines. Define stopping criteria before analysis begins. |
| **Recovery** | Force conclusion immediately. Impose a hard deadline. Select the best available alternative based on current analysis. Flag the reasoning as time-forced. |

### 10.3 Logical Failures

Failures in the formal logical structure of reasoning — errors in how conclusions follow from premises.

#### 10.3.1 False Premise

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process builds on a premise that is not true. The logical structure may be valid, but the foundation is unsound. |
| **Symptoms** | Valid reasoning from false premises produces false conclusions. The reasoning chain appears correct, but the outcome does not match reality. |
| **Root causes** | (1) Unvalidated assumption treated as fact. (2) Outdated Knowledge used despite change. (3) Misinterpreted evidence accepted as premise. (4) Inherited false premise from upstream system. |
| **Detection** | Conclusion contradicts observed reality despite valid logic. Premise validation reveals unsupported statement. |
| **Severity** | Critical — false premises propagate through the entire reasoning chain and contaminate the conclusion. |
| **Prevention** | Validate all premises before use. Distinguish verified from unverified premises. Test premise sensitivity. Require evidence or Knowledge support for each premise. |
| **Recovery** | Identify the false premise. Trace its origin. Replace with validated premise. Re-run reasoning from the corrected foundation. |

#### 10.3.2 Non Sequitur

| Property | Definition |
|----------|-----------|
| **Definition** | The conclusion does not follow from the premises. There is a logical gap in the reasoning chain where the conclusion is unsupported by preceding steps. |
| **Symptoms** | Conclusion appears unrelated to the evidence presented. The Owner cannot follow how the conclusion was reached. Reasoning chain has a discontinuity. |
| **Root causes** | (1) Missing reasoning step that would connect premise to conclusion. (2) Hidden assumption that bridges the gap but is not stated. (3) Cognitive leap without justification. |
| **Detection** | Reasoning chain audit reveals a gap between premises and conclusion. The conclusion introduces concepts not present in the premises. |
| **Severity** | High — the reasoning is incomplete. The conclusion is asserted, not derived. |
| **Prevention** | Require complete reasoning chains. Every conclusion must trace to premises through explicit steps. Audit for logical gaps. |
| **Recovery** | Identify the logical gap. Fill it with explicit reasoning steps. If the gap cannot be filled, flag the conclusion as unsupported. |

#### 10.3.3 Category Error

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process treats concepts from different categories as equivalent or directly comparable when they are not. Apples are compared to oranges. |
| **Symptoms** | Tradeoff evaluation compares incommensurable quantities. Evidence from different domains is combined without accounting for domain differences. |
| **Root causes** | (1) Insufficient domain distinction. (2) Overly abstract reasoning that loses categorical specificity. (3) Incorrect conceptual mapping. |
| **Detection** | Tradeoff evaluation compares quantities that do not share a common measure. Reasoning mixes concepts from different ontological categories. |
| **Severity** | Medium-High — produces comparisons that are technically invalid but may appear reasonable. |
| **Prevention** | Respect ontological boundaries. Ensure comparisons are within the same category. When cross-category comparison is necessary, establish a common evaluation framework. |
| **Recovery** | Identify the category mismatch. Separate the analysis by category. Re-evaluate using appropriate frameworks for each category. |

### 10.4 Knowledge Failures

Failures that originate in the Knowledge System — errors in what the system knows and how it uses Knowledge.

#### 10.4.1 Knowledge Gap

| Property | Definition |
|----------|-----------|
| **Definition** | Reasoning requires Knowledge that the system does not possess. The gap prevents complete or correct reasoning. |
| **Symptoms** | Reasoning proceeds with partial information. Confidence is limited by unknown domain. The system cannot evaluate certain alternatives because the required Knowledge does not exist. |
| **Root causes** | (1) Domain not yet learned. (2) Knowledge has not been validated for the specific context. (3) Knowledge exists but was not retrieved. (4) Knowledge was retired and not replaced. |
| **Detection** | Knowledge retrieval returns no results for a required domain. Context completeness check identifies a Knowledge domain with no entries. |
| **Severity** | Medium-High — gaps reduce confidence and may lead to incomplete recommendations. |
| **Prevention** | Maintain Knowledge coverage map. Identify Knowledge gaps proactively. Trigger Learning when gaps are detected during reasoning. |
| **Recovery** | Flag the Knowledge gap explicitly in the reasoning output. Reduce confidence proportional to gap impact. Trigger Learning to fill the gap. |

#### 10.4.2 Knowledge Conflict

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process encounters conflicting Knowledge items and does not resolve the conflict before proceeding. Reasoning is built on contradictory foundations. |
| **Symptoms** | Reasoning uses contradictory Knowledge items. Conclusion depends on one Knowledge item while ignoring or discounting a conflicting one. |
| **Root causes** | (1) Knowledge conflict not detected during retrieval. (2) Knowledge conflict detected but not resolved. (3) Knowledge System itself has an unresolved conflict. |
| **Detection** | Conflicting Knowledge items are both referenced in the same reasoning process. |
| **Severity** | Critical — reasoning built on contradictory foundations cannot produce reliable conclusions. |
| **Prevention** | Implement Knowledge conflict detection in the retrieval phase. Do not proceed with unresolved conflicts. Require conflict resolution before reasoning advances. |
| **Recovery** | Resolve the Knowledge conflict according to Knowledge System rules (validated over deprecated, newer over older, higher confidence over lower). If unresolvable, flag for Learning and present both perspectives with documented conflict. |

#### 10.4.3 Stale Knowledge

| Property | Definition |
|----------|-----------|
| **Definition** | Reasoning uses Knowledge that was once valid but is no longer current. The Knowledge has not been updated to reflect changed conditions. |
| **Symptoms** | Recommendations are based on outdated assumptions. System does not account for recent changes that should have updated Knowledge. |
| **Root causes** | (1) Knowledge validity period expired. (2) Knowledge not reviewed for continued relevance. (3) Change occurred but Knowledge System was not notified. |
| **Detection** | Knowledge freshness check fails. Knowledge timestamp is older than its validity period for the current context. |
| **Severity** | High — stale Knowledge leads to confidently wrong recommendations. |
| **Prevention** | Implement Knowledge freshness tracking. Require periodic Knowledge review. Detect changes that invalidate existing Knowledge. |
| **Recovery** | Flag stale Knowledge. Retrieve or request updated Knowledge. If update is not available, reduce confidence and flag the gap. |

### 10.5 Evidence Failures

Failures that originate in the Evidence System — errors in how evidence is gathered, evaluated, and weighted.

#### 10.5.1 Weak Evidence

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process relies on evidence that is insufficient in quality, quantity, or relevance to support its conclusion. |
| **Symptoms** | Conclusion is not supported by evidence quality. Evidence does not adequately address the reasoning question. Evidence is outdated, from unreliable sources, or irrelevant. |
| **Root causes** | (1) Evidence not evaluated for quality. (2) Evidence sufficiency not assessed. (3) Conclusion over-extrapolated from limited evidence. (4) Evidence retrieval failed to find stronger evidence. |
| **Detection** | Evidence-to-conclusion gap is visible. Confidence exceeds what evidence quality supports. |
| **Severity** | Medium-High — conclusions are logically structured but empirically unsupported. |
| **Prevention** | Implement evidence sufficiency check before advancing to conclusion. Calibrate confidence to evidence quality. Require minimum evidence quality threshold per reasoning type. |
| **Recovery** | Identify evidence gaps explicitly. Retrieve better evidence if available. Reduce confidence to match actual evidence strength. Flag evidence limitations in the recommendation. |

#### 10.5.2 Missing Evidence

| Property | Definition |
|----------|-----------|
| **Definition** | Critical evidence required for complete reasoning is absent. The reasoning proceeds without information that would significantly affect the conclusion. |
| **Symptoms** | Reasoning acknowledges evidence gaps. Conclusion depends on assumptions that evidence could validate or invalidate. |
| **Root causes** | (1) Evidence not collected. (2) Evidence exists but was not retrieved. (3) Evidence collection is infeasible. (4) The need for evidence was not identified. |
| **Detection** | Evidence checklist for the reasoning type has items that are not collected. Context completeness check identifies evidence gaps. |
| **Severity** | High — missing evidence can lead to wrong conclusions with unwarranted confidence. |
| **Prevention** | Define evidence requirements per reasoning type. Implement evidence completeness check. Flag missing evidence before advancing to conclusion. |
| **Recovery** | If evidence can be collected, defer reasoning until evidence is available. If evidence cannot be collected, flag as permanent gap and reduce confidence accordingly. |

#### 10.5.3 Evidence Misweighting

| Property | Definition |
|----------|-----------|
| **Definition** | Evidence is assigned inappropriate weight in the reasoning process. Strong evidence is discounted or weak evidence is overvalued. |
| **Symptoms** | Conclusion is driven by weak evidence while strong evidence is ignored. The reasoning chain does not reflect actual evidence strength. |
| **Root causes** | (1) Evidence quality not properly assessed. (2) Cognitive bias favoring convenient evidence. (3) Evidence strength criteria incorrectly applied. (4) Confirmation bias affecting weight assignment. |
| **Detection** | Weight assigned to evidence does not correspond to its quality score. Sensitivity analysis reveals conclusion depends on weak evidence. |
| **Severity** | High — systematically misweighted evidence produces systematically skewed recommendations. |
| **Prevention** | Implement evidence quality scoring independent of reasoning. Separate evidence quality assessment from evidence use. Audit evidence weights against quality scores. |
| **Recovery** | Re-evaluate evidence weights. Re-run reasoning with corrected weights. Document the misweighting and correction for Learning. |

### 10.6 Memory Failures

Failures that originate in the Memory System — errors in how past experience is recalled and applied.

#### 10.6.1 Irrelevant Precedent

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process retrieves and applies memories of past situations that are not relevant to the current situation, leading to inappropriate conclusions. |
| **Symptoms** | Recommendation is based on a past situation that does not match current conditions. The Owner notices the precedent is not applicable. |
| **Root causes** | (1) Memory similarity assessment was too broad. (2) Key contextual differences were not identified. (3) Surface similarity masked deeper differences. |
| **Detection** | Comparison of past situation and current situation reveals significant contextual differences not accounted for. |
| **Severity** | Medium — leads to recommendations that may not fit the current situation. |
| **Prevention** | Implement relevance scoring for Memory retrieval. Require contextual similarity assessment before applying precedent. Flag key differences. |
| **Recovery** | Identify the contextual mismatch. Retrieve more relevant precedents or proceed without precedent. Flag the irrelevant retrieval for Memory System improvement. |

#### 10.6.2 Missing Precedent

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process does not retrieve a relevant past experience that would inform or improve the current reasoning. The system fails to learn from history. |
| **Symptoms** | Recommendation does not account for lessons from similar past situations. The same mistake risks being repeated. |
| **Root causes** | (1) Memory retrieval failed to find relevant precedent. (2) Precedent exists but is not indexed for the current context. (3) Precedent was retired or archived and not accessible. |
| **Detection** | Post-hoc analysis reveals a relevant precedent existed but was not used. |
| **Severity** | Medium — missed learning opportunity. May lead to suboptimal recommendations. |
| **Prevention** | Improve Memory retrieval indexing. Expand similarity search scope. Maintain accessibility of important precedents. |
| **Recovery** | If the missing precedent is identified during reasoning, incorporate it. If identified post-hoc, flag for Learning. |

### 10.7 Attention Failures

Failures that originate in the Attention System — errors in what the system considers important.

#### 10.7.1 Wrong Priority

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process addresses a question that is not the most important question. Attention was allocated to a lower-priority stimulus while a higher-priority stimulus was ignored. |
| **Symptoms** | Reasoning output is correct but unimportant. The system focuses on minor issues while major issues go unattended. Owner expresses surprise at what the system prioritized. |
| **Root causes** | (1) Attention allocation error. (2) Priority assessment incorrect. (3) Important stimulus not detected. (4) Owner's actual priorities not reflected in Attention model. |
| **Detection** | Comparison of reasoning processes against Owner's stated priorities reveals misalignment. |
| **Severity** | Medium-High — correct reasoning about the wrong thing is wasted cognition. |
| **Prevention** | Align Attention allocation with Owner priorities. Regularly validate priority model against Owner feedback. Use multiple Attention signals to detect important stimuli. |
| **Recovery** | Re-prioritize Attention allocation. If the reasoning process is already underway, complete it but flag the priority issue. Adjust Attention model. |

#### 10.7.2 Attention Cascade

| Property | Definition |
|----------|-----------|
| **Definition** | A single Attention trigger spawns multiple reasoning processes that interfere with each other, compete for resources, or produce conflicting recommendations. |
| **Symptoms** | Multiple reasoning processes addressing related questions produce contradictory recommendations. Cognitive resources are fragmented across overlapping processes. |
| **Root causes** | (1) Attention trigger interpreted too broadly. (2) Lack of coordination between reasoning processes. (3) No de-duplication of related reasoning needs. |
| **Detection** | Multiple active reasoning processes with overlapping scope. Conflicting recommendations from related processes. |
| **Severity** | Medium — wastes cognitive resources and may confuse the Owner with contradictory outputs. |
| **Prevention** | Implement reasoning process de-duplication. Consolidate related Attention triggers into a single reasoning process with broader scope. Coordinate related processes. |
| **Recovery** | Consolidate overlapping processes. Resolve conflicts between their outputs. Learn to detect related triggers earlier. |

### 10.8 Confidence Failures

Failures in confidence estimation — errors in how the system assesses its own certainty.

#### 10.8.1 Overconfidence

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process expresses higher confidence than the evidence and Knowledge support. Certainty exceeds justification. |
| **Symptoms** | High confidence expressed with weak evidence. Recommendation presented as certain when significant uncertainty exists. Owner acts on false certainty. |
| **Root causes** | (1) Confidence estimation error. (2) Evidence quality overestimated. (3) Knowledge completeness overestimated. (4) Unconscious bias toward certainty. (5) Failure to account for unknown unknowns. |
| **Detection** | Confidence exceeds evidence strength. Historical accuracy at similar confidence levels is measurably lower than expected. Calibration audit reveals systematic overconfidence. |
| **Severity** | High — leads to decisions based on false certainty. Overconfident recommendations are the most dangerous because they invite unquestioning action. |
| **Prevention** | Implement confidence calibration tracking. Compare confidence to actual correctness over time. Require evidence-to-confidence ratio validation. Include uncertainty quantification in every assessment. |
| **Recovery** | Recalibrate confidence immediately. Identify sources of overestimation. Reduce confidence to match evidence support. Flag historical recommendations affected by overconfidence. |

#### 10.8.2 Underconfidence

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process expresses lower confidence than the evidence and Knowledge support. Certainty is less than justification warrants. |
| **Symptoms** | Low confidence despite strong evidence. Recommendation presented hesitantly when it could be confident. Owner may discount valid recommendations due to apparent uncertainty. |
| **Root causes** | (1) Confidence estimation error. (2) Evidence quality underestimated. (3) Excessive caution from past overconfidence corrections. (4) Past errors causing systematic overcorrection. |
| **Detection** | Confidence is lower than evidence strength warrants. Historical accuracy at similar confidence levels is measurably higher than expected. |
| **Severity** | Medium — reduces the impact of valid recommendations. May cause Owner to ignore correct analysis. |
| **Prevention** | Implement confidence calibration tracking. Compare confidence to actual correctness. Calibrate against evidence strength objectively. |
| **Recovery** | Recalibrate confidence upward. Review evidence strength objectively. Document the calibration adjustment. |

#### 10.8.3 Confidence Drift

| Property | Definition |
|----------|-----------|
| **Definition** | Confidence changes during the reasoning process without corresponding changes in evidence, Knowledge, or context. Confidence is unstable without justification. |
| **Symptoms** | Early confidence differs significantly from final confidence without clear reason. Confidence oscillates during the reasoning process. |
| **Root causes** | (1) Non-evidentiary factors affecting confidence (mood, fatigue, noise). (2) Inconsistent confidence estimation criteria. (3) Cognitive bias affecting confidence at different stages. |
| **Detection** | Confidence trajectory shows significant variation without corresponding input changes. |
| **Severity** | Low-Medium — indicates noise in confidence estimation rather than systemic error. |
| **Prevention** | Stabilize confidence estimation criteria. Isolate confidence from non-evidentiary factors. |
| **Recovery** | Identify the source of drift. Apply consistent criteria retroactively. Stabilize confidence estimation method. |

### 10.9 Recommendation Failures

Failures in recommendation construction and presentation — errors in how understanding is communicated.

#### 10.9.1 Recommendation Drift

| Property | Definition |
|----------|-----------|
| **Definition** | The recommendation changes significantly as the reasoning process progresses, without new evidence or Knowledge to justify the change. The system's output is unstable. |
| **Symptoms** | Final recommendation differs significantly from intermediate recommendations without clear justification. The Owner observes inconsistency. |
| **Root causes** | (1) Unstable inputs varying during reasoning. (2) Inconsistent evaluation criteria applied across iterations. (3) Weight drift during tradeoff evaluation. (4) Evaluation criteria changing during the process. |
| **Detection** | Comparison of intermediate recommendations across lifecycle stages shows significant variation without corresponding input changes. |
| **Severity** | Medium — indicates unstable reasoning. Reduces Owner trust in the system. |
| **Prevention** | Log intermediate recommendations at each lifecycle stage. Implement drift detection. Freeze evaluation criteria once tradeoff evaluation begins. |
| **Recovery** | Identify the point of drift. Determine if new information justified the change. If not, stabilize evaluation criteria and re-run reasoning. Document the drift for Learning. |

#### 10.9.2 Inactionable Recommendation

| Property | Definition |
|----------|-----------|
| **Definition** | The recommendation cannot be acted upon by the Owner. It is too vague, too complex, too expensive, or otherwise infeasible to implement. |
| **Symptoms** | Owner cannot determine what action to take based on the recommendation. Recommendation lacks specific guidance. |
| **Root causes** | (1) Recommendation not translated into actionable terms. (2) Constraints not fully evaluated. (3) Owner's implementation capacity not considered. (4) Recommendation too abstract. |
| **Detection** | Owner feedback indicates the recommendation is not actionable. |
| **Severity** | Medium — the reasoning produced understanding but no path to action. |
| **Prevention** | Include actionability as a recommendation quality criterion. Evaluate implementation feasibility before finalizing. Consider Owner's capacity to act. |
| **Recovery** | Translate the recommendation into specific, actionable steps. Identify resources needed for implementation. Reformulate for actionability. |

#### 10.9.3 Owner Misalignment

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process produces a recommendation that does not align with the Owner's goals, preferences, values, or decision-making style. |
| **Symptoms** | Owner rejects recommendation. Owner expresses surprise or disagreement. Recommendation ignores known Owner preferences or past decisions. |
| **Root causes** | (1) Owner preferences not retrieved before reasoning. (2) Owner preferences misinterpreted or stale. (3) Reasoning prioritized other factors (optimization, efficiency) over Owner alignment. (4) Owner's values not captured in the system. |
| **Detection** | Owner feedback indicates misalignment. Owner explicitly disagrees with the recommendation's foundation. |
| **Severity** | High — undermines trust in the system. The most analytically correct recommendation is useless if the Owner will not accept it. |
| **Prevention** | Always retrieve latest Owner preferences before reasoning. Include Owner alignment as a mandatory evaluation criterion. Validate key assumptions about Owner preferences. |
| **Recovery** | Retrieve current Owner preferences. Re-evaluate recommendation with Owner alignment as the primary factor. Present corrected recommendation with alignment explanation. |

### 10.10 Business Failures

Failures that originate in the business context — errors in how business reality is represented and considered.

#### 10.10.1 Context Blindness

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process does not account for the full business context. Critical contextual factors — time of day, season, current events, business conditions — are missing from the analysis. |
| **Symptoms** | Conclusion does not account for relevant conditions. The Owner identifies missing context. Recommendation is inappropriate for the current business situation. |
| **Root causes** | (1) Context not retrieved before reasoning. (2) Context retrieval failed or was incomplete. (3) Relevant context factors not known to exist. (4) Context changed after retrieval but before reasoning completed. |
| **Detection** | Context completeness check fails. Missing context factors are identified. |
| **Severity** | High — context-blind reasoning produces context-blind conclusions that are reliably wrong in important ways. |
| **Prevention** | Implement context completeness checklist per reasoning type. Require context validation before proceeding. Refresh context if reasoning duration exceeds threshold. |
| **Recovery** | Retrieve missing context. Re-evaluate reasoning with complete context. If context cannot be retrieved, flag as incomplete-context reasoning and reduce confidence. |

#### 10.10.2 False Assumption

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process makes an assumption about the business environment that is incorrect, and the conclusion depends on this assumption. |
| **Symptoms** | Conclusion is logically valid given assumptions, but assumptions do not match business reality. The Owner identifies the incorrect assumption. |
| **Root causes** | (1) Assumption not validated against current business data. (2) Assumption inherited from an unreliable or outdated source. (3) Assumption was true when made but business conditions changed. (4) Assumption was never explicitly stated. |
| **Detection** | Assumption sensitivity analysis reveals conclusion depends on an unverified assumption. |
| **Severity** | High — false assumptions propagate through the entire reasoning chain and contaminate all downstream conclusions. |
| **Prevention** | Explicitly flag all assumptions. Distinguish verified from unverified assumptions. Test key assumptions for sensitivity. Validate assumptions against current business data. |
| **Recovery** | Identify the false assumption. Validate or correct it against current business reality. Re-run reasoning with the corrected assumption. |

#### 10.10.3 Constraint Violation

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning process produces a recommendation that violates a known business constraint (budget, regulatory, operational, strategic). |
| **Symptoms** | Recommendation exceeds budget, violates regulations, or conflicts with operational limits. Constraint evaluation fails after recommendation is built. |
| **Root causes** | (1) Constraints not evaluated during reasoning. (2) Constraints changed after evaluation. (3) Constraint misunderstood or misapplied. (4) Constraint intentionally bypassed without documentation. |
| **Detection** | Recommendation fails constraint validation. Post-hoc audit identifies constraint violation. |
| **Severity** | High — violated constraints may lead to regulatory penalties, financial loss, or operational failures. |
| **Prevention** | Evaluate constraints before building recommendations. Re-validate constraints after recommendation is built. Document all constraint evaluations. |
| **Recovery** | Identify the violated constraint. Determine if the constraint was evaluated. If constraints changed, re-evaluate. If constraint was bypassed, flag for Owner approval. If violation is unintentional, re-run reasoning. |

### 10.11 Systemic Failures

Failures that affect the reasoning system as a whole — not isolated to a single reasoning process.

#### 10.11.1 Feedback Loop

| Property | Definition |
|----------|-----------|
| **Definition** | The output of reasoning feeds back into the system and creates a self-reinforcing cycle that amplifies errors or biases over multiple iterations. |
| **Symptoms** | Successive reasoning processes reinforce each other's errors. The system becomes increasingly confident in wrong conclusions over time. Correction becomes harder with each iteration. |
| **Root causes** | (1) Reasoning output used as input to subsequent reasoning without independent validation. (2) Confirmation bias amplified across processes. (3) Learning from incorrect conclusions reinforces the error. |
| **Detection** | Trend analysis shows increasing confidence without increasing evidence quality. Successive recommendations converge on a single view without new supporting evidence. |
| **Severity** | Critical — feedback loops can cause systemic reasoning failure that is self-sustaining and self-reinforcing. |
| **Prevention** | Require independent validation for reasoning outputs that feed into subsequent reasoning. Implement diversity of inputs. Monitor for self-reinforcing patterns. |
| **Recovery** | Break the loop by introducing independent evidence. Reset the affected reasoning chains. Audit the loop's history and correct contaminated Knowledge. |

#### 10.11.2 Cascade Failure

| Property | Definition |
|----------|-----------|
| **Definition** | A failure in one reasoning process propagates to dependent processes, causing a chain of failures across the system. |
| **Symptoms** | A single error in one reasoning process causes failures in multiple downstream processes. The scope of failure expands beyond the original error. |
| **Root causes** | (1) Tight coupling between reasoning processes. (2) Insufficient isolation between processes. (3) Critical flawed output consumed by many processes. (4) No containment mechanisms for reasoning errors. |
| **Detection** | Error propagation tracing shows the original failure spreading to dependent processes. |
| **Severity** | Critical — a single error can cascade through the entire reasoning system. |
| **Prevention** | Implement reasoning process isolation. Contain errors at the process boundary. Validate inputs from other reasoning processes. Implement circuit breakers for dependent chains. |
| **Recovery** | Identify the original failure source. Isolate affected processes. Re-run affected reasoning after the source is corrected. Audit all downstream impacts. |

#### 10.11.3 Resource Exhaustion

| Property | Definition |
|----------|-----------|
| **Definition** | The reasoning system does not have sufficient cognitive resources to complete its required reasoning workload. Processes compete for limited resources and quality degrades. |
| **Symptoms** | Reasoning duration increases. Multiple processes stall. Quality scores decline. Processes are terminated prematurely due to resource pressure. |
| **Root causes** | (1) Excessive concurrent reasoning demands. (2) Insufficient cognitive capacity for current workload. (3) Inefficient reasoning consuming excessive resources. (4) No resource management or prioritization. |
| **Detection** | Resource utilization exceeds sustainable threshold. Process completion rate drops. |
| **Severity** | High — systemic degradation affects all reasoning quality. |
| **Prevention** | Implement reasoning resource budgeting. Prioritize reasoning processes by importance. Set maximum concurrent processes. Implement load shedding for low-priority reasoning. |
| **Recovery** | Shed low-priority reasoning processes. Complete highest-priority processes first. Escalate if resource pressure persists. Review workload distribution. |

### 10.12 Detection Framework

Every failure mode must be detectable through systematic monitoring. The following detection methods apply across all failure categories.

| Detection Method | Applies To | Mechanism |
|-----------------|------------|-----------|
| **Lifecycle monitoring** | All failures | Track stage duration, re-entry count, completion rate. Deviations from expected patterns indicate failure. |
| **Evidence distribution analysis** | Cognitive, Evidence failures | Analyze evidence support/contradict ratio. Skewed distribution indicates bias. |
| **Confidence calibration tracking** | Confidence failures | Compare confidence to outcome correctness over time. Systematic deviation indicates calibration error. |
| **Reasoning chain audit** | Logical failures | Trace premises to conclusions. Identify cycles, gaps, discontinuities. |
| **Context completeness check** | Business failures | Verify all required context factors are present before proceeding. |
| **Constraint validation** | Business failures | Verify recommendation against all active constraints before output. |
| **Cross-process correlation** | Systemic failures | Detect patterns across multiple reasoning processes that indicate systemic issues. |
| **Owner feedback monitoring** | Recommendation failures | Track acceptance, rejection, and satisfaction rates. |
| **Knowledge freshness check** | Knowledge failures | Verify Knowledge currency against validity periods. |
| **Memory relevance scoring** | Memory failures | Score memory similarity and flag low-relevance retrievals. |

### 10.13 Prevention Framework

| Prevention Layer | Measures |
|-----------------|----------|
| **Structural** | Lifecycle stage gates, mandatory checks at each transition, minimum requirements per reasoning type. |
| **Procedural** | Checklists per reasoning type, pre-reasoning validation, post-reasoning audit. |
| **Monitoring** | Real-time detection of failure patterns, trend analysis, anomaly detection. |
| **Calibration** | Regular confidence calibration, evidence quality recalibration, assumption validation cycles. |
| **Learning** | Post-mortem analysis of failures, systemic improvement from failure patterns, Knowledge updates. |
| **Constitutional** | Policy enforcement, invariant verification, constitutional audit. |

### 10.14 Recovery Framework Summary

| Recovery Type | Description | Applied To |
|---------------|-------------|------------|
| **Self-correction** | Automatic recovery within the same reasoning process. | Overconfidence, Underconfidence, Weak Evidence, False Assumption |
| **Re-run** | Abandon current process and restart with corrections. | Circular Reasoning, Knowledge Conflict, False Premise |
| **Escalation** | Involve the Owner or higher authority. | Reasoning Collapse, Owner Misalignment, Constraint Violation |
| **Isolation** | Contain the failure and prevent propagation. | Cascade Failure, Feedback Loop |
| **Systemic reset** | Recalibrate system-wide parameters. | Resource Exhaustion, Cascade Failure |

## Part 11 — Recovery Framework

### 11.1 Recovery Philosophy

Recovery in the Reasoning Engine is not failure handling. It is the constitutional mechanism by which the system maintains its cognitive integrity when reasoning fails. Recovery is not optional. Recovery is not exceptional. Recovery is an intrinsic property of a resilient reasoning system.

Every failure mode has a corresponding recovery path. Recovery is not a single action — it is a lifecycle with defined stages, triggers, strategies, and learning outcomes.

### 11.2 Recovery Lifecycle

Every recovery event progresses through the following stages:

| Stage | Purpose | Entry | Exit |
|-------|---------|-------|------|
| **Detection** | A failure is identified through monitoring, audit, or Owner feedback. | Failure condition met. | Failure confirmed and classified. |
| **Assessment** | The failure is evaluated for severity, scope, root cause, and impact. | Failure confirmed. | Recovery strategy selected. |
| **Containment** | The failure is contained to prevent propagation. | Strategy selected. | Failure isolated. |
| **Resolution** | The recovery strategy is executed. | Failure contained. | Failure resolved or escalated. |
| **Verification** | The resolution is verified for correctness. | Resolution executed. | Resolution confirmed or rejected. |
| **Learning** | The failure and recovery are analyzed for systemic improvement. | Resolution confirmed. | Learning documented and applied. |
| **Closure** | The recovery event is formally closed. | Learning completed. | Recovery archived. |

### 11.3 Recovery Triggers

Recovery is triggered by the following conditions:

| Trigger Category | Trigger Condition | Source |
|------------------|-------------------|--------|
| **Lifecycle violation** | Stage duration exceeded. Stage re-entry exceeds limit. | Lifecycle monitoring |
| **Quality threshold** | Quality dimension score below threshold. | Quality assessment |
| **Consistency failure** | Internal contradiction detected. External contradiction with Knowledge. | Consistency checking |
| **Confidence anomaly** | Overconfidence detected. Underconfidence detected. Confidence drift. | Confidence assessment |
| **Evidence failure** | Evidence quality below threshold. Evidence gap critical. Evidence conflict. | Evidence evaluation |
| **Knowledge failure** | Knowledge conflict. Stale Knowledge. Knowledge gap. | Knowledge retrieval |
| **Constraint violation** | Recommendation violates constraint. No alternative survives constraints. | Constraint evaluation |
| **Owner feedback** | Owner rejects recommendation. Owner indicates misalignment. | Owner interaction |
| **Systemic anomaly** | Resource exhaustion. Cascade pattern detected. Feedback loop detected. | System monitoring |
| **External signal** | Business conditions change significantly. New evidence invalidates reasoning. | Business Pulse |

### 11.4 Recovery Strategies

The Recovery Framework defines five primary strategies. Each strategy is appropriate for specific failure types and severities.

#### 11.4.1 Self-Correction

The reasoning process automatically corrects the failure within its current lifecycle without restarting.

| Property | Definition |
|----------|-----------|
| **When to use** | Minor failures, quality deviations, confidence calibration issues. Failures that can be corrected without invalidating work already done. |
| **How it works** | The process returns to an earlier lifecycle stage, corrects the issue, and proceeds forward again. |
| **Success criteria** | The corrected process completes successfully with all quality dimensions above threshold. |
| **Failure escalation** | If self-correction fails after maximum attempts, escalate to Retry or Escalation. |
| **Applicable failures** | Overconfidence, Underconfidence, Weak Evidence, Evidence Misweighting, Confidence Drift, Recommendation Drift |
| **Owner visibility** | Low — Owner is notified of correction but does not need to intervene. |

#### 11.4.2 Retry

The reasoning process is abandoned and restarted with corrected inputs, parameters, or scope.

| Property | Definition |
|----------|-----------|
| **When to use** | Failures caused by incorrect inputs, stale Knowledge, false premises, or inappropriate scope. The foundation of the reasoning was flawed. |
| **How it works** | Archive the current process. Identify and correct the root cause. Start a new reasoning process with corrected foundations. |
| **Success criteria** | The new process completes successfully. Root cause is resolved. |
| **Failure escalation** | If Retry fails (same failure occurs), escalate to Escalation. |
| **Applicable failures** | False Premise, Stale Knowledge, Knowledge Gap, Missing Evidence, Irrelevant Precedent, Context Blindness |
| **Owner visibility** | Medium — Owner is notified of the retry and the root cause correction. |

#### 11.4.3 Escalation

The failure is escalated to a higher authority — either a more capable reasoning process or the Owner.

| Property | Definition |
|----------|-----------|
| **When to use** | Failures that cannot be resolved within the current reasoning framework. Requires judgment beyond the system's current capability. |
| **How it works** | Package the failure with full context: what failed, why, what was attempted, what is needed. Escalate to the next authority level. |
| **Success criteria** | The escalated authority resolves the failure or provides guidance for resolution. |
| **Failure escalation** | If escalation is rejected or unresolved, escalate to the Owner. The Owner is the final escalation authority. |
| **Applicable failures** | Reasoning Collapse, Owner Misalignment, Constraint Violation, Cascade Failure, Feedback Loop |
| **Owner visibility** | High — Owner is actively involved in resolution. |

#### 11.4.4 Rollback

The reasoning output is retracted or superseded. The system returns to the state before the reasoning occurred.

| Property | Definition |
|----------|-----------|
| **When to use** | Failures discovered after the reasoning has completed and potentially influenced decisions. The recommendation was wrong and may have been acted upon. |
| **How it works** | Retract the recommendation. Notify all processes that consumed the reasoning output. Trigger corrective action if a decision was made based on the flawed reasoning. |
| **Success criteria** | All downstream impacts are identified and addressed. The flawed reasoning is marked as superseded. |
| **Failure escalation** | If downstream impact cannot be contained, escalate for Owner intervention. |
| **Applicable failures** | Reasoning Collapse (after incomplete output), Cascade Failure, Feedback Loop, False Assumption (discovered post-hoc) |
| **Owner visibility** | Critical — Owner must be notified immediately of any rollback that affects decisions or actions. |

#### 11.4.5 Human Intervention

The Owner directly intervenes in the reasoning process.

| Property | Definition |
|----------|-----------|
| **When to use** | Failures that require human judgment, values, or authority. Situations where the system cannot determine the correct path. |
| **How it works** | Present the failure to the Owner with full context. Provide the system's analysis and recommended resolution. The Owner decides. The system implements the Owner's decision. |
| **Success criteria** | Owner provides resolution guidance. |
| **Failure escalation** | If Owner cannot resolve, document the unresolvable failure and flag for systemic improvement. |
| **Applicable failures** | Owner Misalignment, Constraint Violation (requiring relaxation), Ethical dilemmas, Strategic conflicts |
| **Owner visibility** | Full — Owner is the active decision-maker. |

### 11.5 Escalation Ladder

When a failure cannot be resolved at one level, it escalates to the next level in the hierarchy:

```
Level 1: Self-Correction (automatic, within process)
Level 2: Retry (automatic, new process instance)
Level 3: Escalation to Composite Reasoning (if applicable)
Level 4: Escalation to Meta-Reasoning (system evaluation)
Level 5: Escalation to Owner (human judgment)
Level 6: Systemic Change (constitutional amendment, policy update)
```

### 11.6 Rollback Protocol

When reasoning output must be retracted:

1. **Identify scope.** Determine which decisions, actions, and downstream reasoning processes were affected by the flawed output.
2. **Contain propagation.** Notify all affected processes that their inputs are potentially invalid.
3. **Assess impact.** For each affected process, determine whether its output is also invalidated.
4. **Execute rollback.** Retract the flawed recommendation. If decisions were made, flag them for review.
5. **Trigger corrective action.** If actions were taken based on the flawed reasoning, recommend corrective actions.
6. **Document.** Record the rollback with full traceability from original reasoning through all downstream impacts.
7. **Learn.** Analyze why the failure was not detected before the output was used.

### 11.7 Retry Protocol

When a reasoning process must be restarted:

1. **Archive current process.** Preserve the failed process for audit and learning.
2. **Diagnose root cause.** Determine exactly what caused the failure.
3. **Correct inputs.** Fix the inputs (evidence, Knowledge, context, assumptions) that led to the failure.
4. **Adjust parameters.** If necessary, adjust reasoning parameters (scope, time limits, quality thresholds).
5. **Start new process.** Begin a new reasoning process with corrected foundations.
6. **Verify.** Confirm the new process does not repeat the same failure.
7. **Limit retry attempts.** Maximum retry attempts = 3. After 3 failures, escalate.

### 11.8 Human Intervention Protocol

When the Owner must intervene:

1. **Prepare context.** Package the failure with: what was being reasoned about, what failed, what was attempted, what is needed from the Owner.
2. **Present clearly.** Present the situation in language the Owner understands. Avoid cognitive terminology. Use business terms.
3. **Provide options.** Offer the Owner clear options for resolution, with tradeoffs of each.
4. **Respect sovereignty.** The Owner's decision is final. The system does not argue with the Owner's choice.
5. **Implement.** Execute the Owner's decision.
6. **Document.** Record the intervention for learning and audit.

### 11.9 Learning After Recovery

Every recovery event generates learning:

| Learning Type | What Is Learned | Where It Goes |
|---------------|-----------------|---------------|
| **Root cause pattern** | What types of failures occur repeatedly. | Knowledge System — failure pattern Knowledge |
| **Recovery effectiveness** | Which recovery strategies work for which failure types. | Meta-Reasoning — recovery effectiveness Knowledge |
| **Prevention opportunity** | What could have prevented the failure. | Policy System — policy improvement candidates |
| **Detection gap** | Why the failure was not detected earlier. | Monitoring System — detection improvement |
| **Systemic vulnerability** | What the failure reveals about system weaknesses. | Knowledge System — vulnerability Knowledge |
| **Owner preference insight** | What the Owner's intervention reveals about their values. | Owner preference records — refinement |

### 11.10 Recovery Framework Principles

1. **Automatic recovery first.** The system attempts automatic recovery before involving the Owner. Owner attention is scarce and must be respected.
2. **Owner override is always available.** No automatic recovery mechanism can block Owner intervention. The Owner can override any recovery decision.
3. **Recovery generates learning.** Every recovery event is analyzed for systemic improvement. A failure without learning is a failure wasted.
4. **Multiple strategies may combine.** Recovery strategies are composable. Self-correction may be followed by escalation if it fails.
5. **Prolonged failure requires Owner notification.** If recovery fails after maximum attempts, the Owner is informed.
6. **Recovery does not erase history.** The original reasoning and all recovery attempts are preserved for audit.
7. **Speed of recovery is proportional to failure severity.** Critical failures require faster recovery with more aggressive strategies.
8. **Recovery is part of reasoning quality.** A system that recovers well is higher quality than one that never fails but cannot recover.

#### Example 1: Sales Decline Diagnosis

| Aspect | Detail |
|--------|--------|
| **Scenario** | Tuesday dinner sales are 30% below the same day last week and 25% below forecast. |
| **Attention trigger** | Operational Attention — sales deviation threshold exceeded. |
| **Reasoning type** | Diagnostic Reasoning (primary). Operational Reasoning (secondary). |
| **Question** | "Why are Tuesday dinner sales down and what should we do?" |
| **Context** | Tuesday, dinner service, mid-month, no holidays, no special events. Weather: rainy but not unusual. |
| **Evidence gathered** | Sales data by hour, menu item, section, server. Customer count. Average check size. Reservation data. Walk-in data. |
| **Knowledge retrieved** | Seasonal patterns of Tuesday sales. Causal Knowledge: weather impact on dine-in, competitor promotions, local events. |
| **Memory retrieved** | Similar sales declines in past 12 months. Previous causes and outcomes. |
| **Hypotheses generated** | (1) Weather keeping customers home. (2) New competitor promotion. (3) Service issue affecting reputation. (4) Menu fatigue — regulars looking for variety. (5) Random fluctuation — normal variance. |
| **Alternatives generated** | (1) Monitor — do nothing yet. (2) Investigate competitor activity. (3) Review recent reviews for service complaints. (4) Offer mid-week promotion. (5) Check POS for specific item decline patterns. |
| **Constraints evaluated** | Budget for promotion limited. Staff capacity for additional marketing. Time: need to act before weekend. |
| **Tradeoffs evaluated** | Cost of promotion vs. potential sales recovery. Risk of overreacting to normal variance vs. risk of missing real decline. |
| **Consistency check** | Patterns from Memory align with current data. No contradictions. Hypotheses cover the possibility space. |
| **Confidence** | Moderate (6/10) — multiple hypotheses could explain the decline. Evidence not yet sufficient to isolate cause. |
| **Recommendation** | (1) Compare sales data to competitor opening dates — if a new competitor opened recently, this is likely the cause. (2) If no competitor change, check recent review patterns. (3) If neither shows a cause, implement a mid-week promotion and monitor. |
| **Outcome** | System identifies new competitor opened two weeks ago. Sales decline correlates with competitor opening date. Recommendation accepted. |

#### Example 5: Promotion Analysis

## Part 13 — Constitutional Principles

### 13.1 Constitutional Framework

The Reasoning Engine operates under the supreme authority of the Constitution of Restaurant OS. Every principle, invariant, policy, and recovery mechanism derives from constitutional authority. Reasoning has no independent authority — it serves the Constitution by serving the business.

### 13.2 Sovereignty Principles

#### Principle 1: Owner Sovereignty

Reasoning advises. The Owner decides. No recommendation, regardless of analytical strength, overrides the Owner's authority. Reasoning that presents itself as directive rather than advisory violates the constitutional order. The Owner may accept, reject, or modify any recommendation without justification.

#### Principle 2: Business Purpose

Every reasoning process must serve a business outcome. Reasoning for its own sake is cognitive waste. The measure of reasoning quality is ultimately whether it produces better business decisions, not whether it is analytically elegant.

#### Principle 3: Predictability

Reasoning must produce consistent, reliable understanding. The same inputs must produce the same conclusions. Random variation, unstable evaluation criteria, and undisciplined reasoning violate predictability. The Owner must be able to anticipate how the system will reason about familiar situations.

#### Principle 4: Adaptability

While reasoning must be predictable, it must also adapt to changing conditions. When context changes, reasoning must change accordingly. Predictability without adaptability is rigidity. Adaptability without predictability is chaos. Reasoning must balance both.

### 13.3 Truth Principles

#### Principle 5: Evidence Fidelity

Reasoning must respect the truthfulness of evidence. Evidence cannot be selectively ignored, distorted, or manufactured to fit a preferred conclusion. Cherry-picking is not reasoning — it is rationalization. Every evidence item must be given its proper weight regardless of whether it supports or contradicts the emerging conclusion.

#### Principle 6: Uncertainty Honesty

Reasoning must know what it does not know. Certainty is earned, not assumed. Every recommendation must communicate what is known, what is uncertain, and what is unknown. Reasoning that presents uncertainty as certainty is not reasoning — it is pretending. The Owner must be able to trust that the system's confidence reflects actual confidence, not performed confidence.

#### Principle 7: Knowledge Respect

Validated Knowledge constrains reasoning. If Knowledge says X is true, reasoning cannot conclude not-X without explicitly resolving the contradiction. Reasoning does not override Knowledge. Reasoning extends Knowledge through valid application to new situations. Knowledge that has been earned through the Knowledge System's validation lifecycle takes precedence over unvalidated hypotheses.

#### Principle 8: Intellectual Honesty

Reasoning must acknowledge its own limitations. Assumptions must be explicit. Gaps must be flagged. Uncertainty must be communicated. Reasoning that hides its weaknesses is not serving the Owner — it is serving its own appearance of competence. Intellectual honesty is the foundation of trust between the system and the Owner.

### 13.4 Structural Principles

#### Principle 9: Traceability

Every reasoning process must be traceable from its originating Attention allocation through every lifecycle stage to its final recommendation and disposition. Traceability is not optional — it is constitutional. Without traceability, reasoning cannot be audited, learned from, or trusted.

#### Principle 10: Completeness

Every reasoning process must complete the full lifecycle appropriate to its type. Skipping stages, cutting corners, and premature conclusions violate completeness. A reasoning process that has not evaluated alternatives, assessed tradeoffs, checked consistency, and estimated confidence has not reasoned — it has guessed.

#### Principle 11: Explainability

Every recommendation must include an explanation chain that the Owner can understand. If the system cannot explain how it reached a conclusion, it has not reasoned — it has produced an output without understanding. Explainability is not a feature. It is the definition of reasoning made manifest.

#### Principle 12: Auditability

Every reasoning process must produce a complete audit trail including all inputs, all lifecycle transitions, all alternatives considered, all tradeoffs evaluated, and the final recommendation. Audit trails are immutable after the process reaches Retired. The audit trail is the constitutional record of reasoning.

### 13.5 Quality Principles

#### Principle 13: Confidence Calibration

Confidence must be calibrated to evidence strength, Knowledge completeness, and context coverage. Overconfidence and underconfidence are both constitutional failures. The system must track its confidence accuracy over time and recalibrate when systematic deviation is detected.

#### Principle 14: Consistency

Reasoning must be internally consistent (no contradictions within the reasoning chain) and externally consistent (no contradictions with established Knowledge). Inconsistency is a reasoning failure. Consistency checking is a mandatory lifecycle stage. No reasoning process may complete without passing consistency checks.

#### Principle 15: Depth Proportionality

The depth of reasoning must be proportional to the importance and complexity of the question. Routine operational questions deserve efficient reasoning. Strategic questions deserve thorough reasoning. Emergency situations deserve rapid reasoning. Depth must match need — neither insufficient nor excessive.

#### Principle 16: Alternative Sufficiency

Every reasoning process must consider at least two genuinely different alternatives. "Do nothing" is always a valid alternative. Alternatives must be feasible within business constraints and must be presented to the Owner or Decision system. A recommendation without alternatives is not a recommendation — it is an instruction.

### 13.6 Boundary Principles

#### Principle 17: Reasoning Does Not Create Knowledge

Reasoning cannot create Knowledge. Reasoning can propose Knowledge candidates, identify Knowledge gaps, and flag contradictions — but the transformation from reasoning output to Knowledge requires the Knowledge System's validation lifecycle. Shortcutting this process violates the constitutional separation between Reasoning and Knowledge.

#### Principle 18: Reasoning Does Not Decide

Reasoning prepares decisions. It does not execute them. The constitutional boundary between Reasoning and Decision is absolute. Reasoning that attempts to act directly bypasses the separation between cognition and execution. Recommendations are advisory. Decisions are the Owner's domain.

#### Principle 19: Reasoning Does Not Replace Attention

Attention determines what matters. Reasoning processes what matters. Reasoning cannot override Attention's selection. If Attention has not selected a stimulus, reasoning does not process it. The constitutional flow is: Attention triggers Reasoning. Not the reverse.

#### Principle 20: Reasoning Does Not Replace Observation

Reasoning uses observations as inputs. It does not generate observations. If an observation is missing, reasoning must flag the gap — not fabricate the missing observation. The boundary between Observation and Reasoning is the boundary between perception and understanding.

### 13.7 Resilience Principles

#### Principle 21: Recovery is Constitutional

The ability to recover from failure is not optional — it is a constitutional requirement of the Reasoning Engine. A system that cannot recover from failure is not resilient. Every failure mode must have a defined recovery path. Recovery is not an afterthought. It is a designed property.

#### Principle 22: Learning From Failure

Every reasoning failure must generate learning. A failure that does not produce learning is a failure wasted. The system must analyze failures for root causes, recovery effectiveness, and prevention opportunities. Learning from failure is how the system improves.

#### Principle 23: Graceful Degradation

When reasoning cannot achieve its ideal quality, it must degrade gracefully rather than fail catastrophically. Partial understanding with documented limitations is better than no understanding. Graceful degradation preserves the system's usefulness even under adverse conditions.

#### Principle 24: Error Containment

Failures in one reasoning process must not propagate to other processes. Each process must be isolated to contain its errors. Cascade failures indicate a systemic vulnerability that must be addressed at the constitutional level. Error containment is a structural requirement.

### 13.8 Ethical Principles

#### Principle 25: Safety Priority

Safety considerations override all other reasoning factors. Reasoning about safety-critical situations must include mandatory safety checks. No recommendation may violate health, safety, or regulatory requirements. If reasoning identifies a safety risk, the recommendation must include a mitigation plan. Safety is absolute.

#### Principle 26: Fairness

Reasoning must not systematically disadvantage any stakeholder group — customers, staff, suppliers, or the Owner. Recommendations must consider the distribution of costs and benefits across stakeholders. Unfair reasoning is not good reasoning, regardless of its analytical quality.

#### Principle 27: Transparency to the Owner

The Owner has the right to inspect any aspect of any reasoning process at any time. There are no hidden reasoning stages, secret evaluation criteria, or opaque confidence calculations. Full transparency is a constitutional right of the Owner.

#### Principle 28: Human Primacy

The system serves the human. Not the reverse. The Owner's judgment, values, and experience take precedence over the system's analysis. The system may challenge the Owner's assumptions with evidence and reasoning, but it must ultimately respect the Owner's authority. Human primacy is the foundation of the entire constitutional order.

### 13.9 Constitutional Limitations on Reasoning

| Limitation | Rationale |
|------------|-----------|
| Reasoning must not invent facts. | Reasoning works with what it is given. Fabrication violates truthfulness. |
| Reasoning must not replace Attention. | Attention decides what matters. Reasoning processes what matters. |
| Reasoning must not replace Knowledge. | Knowledge constrains reasoning. Reasoning does not override validated Knowledge. |
| Reasoning must not execute actions. | Reasoning prepares decisions. Decisions lead to actions. Reasoning does not act. |
| Reasoning must not guarantee outcomes. | Reasoning estimates confidence. It does not promise results. |
| Reasoning must not be infinite. | All reasoning must complete within its time window. |
| Reasoning must not be opaque. | The reasoning chain must be fully explainable to the Owner. |
| Reasoning must not create Knowledge. | Reasoning proposes. The Knowledge System validates. |
| Reasoning must not contradict validated Knowledge without justification. | Valid Knowledge is earned and must be respected. |
| Reasoning must not violate ethical principles. | Ethical operation is a constitutional requirement. |

### 13.10 Constitutional Role of Reasoning

Reasoning occupies a unique constitutional position: it is the bridge between all upstream cognition and all downstream action.

| Constitutional Requirement | Reasoning Role |
|---------------------------|---------------|
| The system must understand its business. | Reasoning transforms information into understanding. |
| The system must explain its understanding. | Reasoning produces explanations. Every recommendation includes a reasoning chain. |
| The system must recommend action. | Reasoning produces recommendations. Every completed process generates at least one recommendation. |
| The system must respect the Owner. | Reasoning advises. The Owner decides. Recommendations are advisory, not directive. |
| The system must learn. | Reasoning identifies Knowledge gaps, contradictions, and learning opportunities. |
| The system must be transparent. | Reasoning is fully explainable. Every step can be traced. |
| The system must be resilient. | Every failure has a recovery path. Failure generates learning. |
| The system must be predictable. | Same inputs produce same conclusions. Evaluation criteria are stable. |
| The system must be truthful. | Evidence is respected. Uncertainty is communicated. Assumptions are explicit. |
| The system must serve the business. | Every reasoning process contributes to business outcomes. |

## Appendix F: Quality Checklist

### F.1 Pre-Reasoning Quality Checks

| Check | Description | Criteria |
|-------|-------------|----------|
| **Question clarity** | Is the reasoning question clearly defined? | Question is specific, answerable, and scoped. |
| **Context completeness** | Is the required context available? | All context factors for the reasoning type are present. |
| **Evidence availability** | Is sufficient evidence available? | Minimum evidence threshold met for the reasoning type. |
| **Knowledge availability** | Is required Knowledge accessible? | Relevant Knowledge exists or gaps are flagged. |
| **Trigger validity** | Is the Attention allocation valid? | Trigger is genuine, not noise. |
| **Scope definition** | Is the scope clearly bounded? | Boundaries are defined. What is in/out of scope. |

### F.2 During-Reasoning Quality Checks

| Check | Description | Criteria |
|-------|-------------|----------|
| **Hypothesis diversity** | Are hypotheses genuinely different? | At least 2 hypotheses from different explanatory categories. |
| **Alternative sufficiency** | Are enough alternatives generated? | Minimum 2 alternatives including do-nothing. |
| **Constraint evaluation** | Are all relevant constraints applied? | All constraint types (regulatory, financial, operational, strategic, safety) checked. |
| **Tradeoff thoroughness** | Are tradeoffs fully evaluated? | Both positive and negative outcomes for each alternative. |
| **Consistency verification** | Is the reasoning chain consistent? | No internal contradictions. Externally consistent with Knowledge. |
| **Assumption documentation** | Are all assumptions documented? | Every assumption is explicit and categorized as verified or unverified. |
| **Uncertainty identification** | Are uncertainty sources identified? | Epistemic, aleatory, model, and measurement uncertainty documented. |

### F.3 Post-Reasoning Quality Checks

| Check | Description | Criteria |
|-------|-------------|----------|
| **Recommendation actionability** | Can the Owner act on the recommendation? | Specific, feasible, resourced. |
| **Confidence calibration** | Is confidence appropriate to evidence? | Confidence matches evidence strength and coverage. |
| **Explanation completeness** | Is the reasoning chain fully explained? | Every step from premises to conclusion is documented. |
| **Owner alignment** | Does the recommendation align with Owner? | Consistent with known Owner preferences and values. |
| **Risk identification** | Are risks of the recommendation identified? | Likelihood and impact documented. Mitigations proposed. |
| **Alternative presentation** | Are alternatives presented? | All alternatives (not just the recommended one) are included. |
| **Outcome projection** | Are expected outcomes documented? | Realistic projection of what will happen if recommendation is followed. |
