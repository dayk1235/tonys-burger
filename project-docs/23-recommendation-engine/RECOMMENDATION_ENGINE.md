# RECOMMENDATION ENGINE — Canonical Specification of Recommendation in Restaurant OS

**Version:** 1.0
**Status:** Constitutional Specification
**Module:** 23-recommendation-engine
**Layer:** Cognitive Architecture — Expression Layer — Output

## Preamble

This document is the canonical specification of Recommendation inside Restaurant OS.

It is not a notification system. It is not a reporting system. It is not an AI assistant. It is not a dashboard. It is not a decision. It is not a command.

The Recommendation Engine is the first engine that transforms cognition into business value.

Observation understands reality. Memory preserves reality. Knowledge validates reality. Attention selects reality. Reasoning explains reality. Recommendation proposes how reality should change.

Recommendation is the bridge between intelligence and action.

It exists before Decision. Decision chooses. Recommendation proposes. This distinction is constitutional and must never be violated.

Every Recommendation Engine output is a proposal. Every proposal is justified. Every justification is transparent. Every recommendation respects Owner sovereignty. Every recommendation can be accepted, rejected, deferred, or ignored — and the system learns from all four outcomes.

## Part 1 — Philosophy of Recommendation

### 1.1 What Recommendation Is

Recommendation is the cognitive process by which Restaurant OS proposes how reality should change.

Recommendation takes reasoning, knowledge, evidence, memory, prediction, and business context — and produces proposals for human consideration.

Recommendation does not decide. Recommendation does not execute. Recommendation does not command. Recommendation informs, suggests, proposes, and advises.

The distinction between Recommendation and Decision is the most important constitutional boundary in Restaurant OS.

### 1.2 Why Recommendation Exists

Recommendation exists because cognition without action has no business value.

Observation detects. Pattern recognizes. Evidence validates. Memory retains. Knowledge understands. Attention prioritizes. Reasoning explains. But none of these produce business value. Business value only emerges when cognition leads to action — and action requires human decision.

The Recommendation Engine is the bridge. It transforms "what is happening" into "what should happen."

Without Recommendation, Restaurant OS would be the most intelligent observer in the world — seeing everything, understanding everything, but never suggesting anything. It would be a perfect mirror that never speaks.

### 1.3 Core Philosophical Principles

**REC-PHI-001 — The Bridge Principle.** Recommendation bridges understanding and action. Without recommendation, intelligence has no business impact.

**REC-PHI-002 — The Proposal Principle.** Every recommendation is a proposal. Never a command. Never an instruction. Never a demand.

**REC-PHI-003 — The Justification Principle.** Every recommendation must include its justification. Reasoning without justification is not recommendation — it is noise.

**REC-PHI-004 — The Sovereignty Principle.** The Owner always has the final authority. Every recommendation must be rejectable without consequence.

**REC-PHI-005 — The Transparency Principle.** The system must explain why it recommended something, including the evidence, confidence, and alternatives considered.

**REC-PHI-006 — The Humility Principle.** Every recommendation carries uncertainty. Confidence must be communicated honestly. Overconfidence is a constitutional violation.

**REC-PHI-007 — The Silence Principle.** Not every situation requires a recommendation. Silence is a valid state. Recommending nothing is sometimes the best recommendation.

**REC-PHI-008 — The Learning Principle.** Every recommendation — accepted, rejected, deferred, or ignored — is a learning opportunity. The system must learn from all outcomes.

**REC-PHI-009 — The Timing Principle.** A correct recommendation at the wrong time is a bad recommendation. Timing is part of recommendation quality.

**REC-PHI-010 — The Narrative Principle.** Recommendations must be communicated as business narratives, not technical data. The Owner thinks in stories, not spreadsheets.

### 1.4 Distinctions

Recommendation is different from the following cognitive entities. Each distinction is constitutional and must never be blurred.

#### Recommendation vs Decision

| Property | Recommendation | Decision |
|----------|---------------|----------|
| **What it is** | A proposal for action. A suggestion presented for human consideration. | A commitment to action. A binding choice that changes the restaurant. |
| **Authority** | None. Can be accepted, rejected, deferred, ignored. | Absolute. A decision changes reality. |
| **Language** | Advisory: "I suggest," "Consider," "An option is." | Committal: "We will," "The decision is," "Proceed with." |
| **Error mode** | Wrong suggestion. Owner ignores it. | Wrong action. Business suffers. |
| **Reversal** | Rejection requires no process. Simply do not accept. | Reversal requires a new decision. |
| **Responsibility** | The system is responsible for the quality of the suggestion. | The Owner is responsible for the decision. |

**Why they are different:** Recommendation proposes. Decision commits. The Recommendation Engine must never blur this line.

#### Recommendation vs Suggestion

| Property | Suggestion | Recommendation |
|----------|-----------|---------------|
| **Scope** | Narrow. A specific idea for a specific moment. | Broad. Considers full business context, evidence, and alternatives. |
| **Depth** | Surface-level. "You could try X." | Deep. Includes reasoning, evidence, confidence, alternatives, impact analysis. |
| **Persistence** | Ephemeral. Said once, forgotten. | Tracked. The system remembers, learns, and evolves its recommendations. |
| **Responsibility** | None. Just an idea. | High. The system justifies and stands behind its recommendations. |

**Why they are different:** A suggestion is casual. A recommendation is a constitutional cognitive act.

#### Recommendation vs Advice

| Property | Advice | Recommendation |
|----------|--------|---------------|
| **Source** | Often external. From a person, book, or experience. | Internal. Generated by the system's cognitive process. |
| **Context** | General. May apply to many situations. | Specific. Tailored to current business context. |
| **Evidence** | May be anecdotal. | Supported by system evidence, patterns, and knowledge. |

**Why they are different:** Advice is general wisdom. Recommendation is contextual cognition.

#### Recommendation vs Prediction

| Property | Prediction | Recommendation |
|----------|-----------|---------------|
| **What it is** | A forecast of a future state. | A proposal for action. |
| **Question it answers** | "What will happen?" | "What should we do?" |
| **Certainty** | Probabilistic. | Advisory. |
| **Action orientation** | Passive. Describes the future. | Active. Proposes changing the future. |

**Why they are different:** Prediction tells you what is coming. Recommendation tells you what to do about it.

#### Recommendation vs Planning

| Property | Planning | Recommendation |
|----------|----------|---------------|
| **Scope** | Sequences multiple actions over time. | Usually a specific action or decision. |
| **Granularity** | Detailed steps, resources, timeline. | Conceptual. What, why, and expected outcome. |
| **Output** | A plan: ordered actions with dependencies. | A proposal: a suggested action with justification. |

**Why they are different:** Planning sequences execution. Recommendation proposes the direction.

#### Recommendation vs Instruction

| Property | Instruction | Recommendation |
|----------|------------|---------------|
| **Tone** | Directive. "Do X." | Advisory. "Consider X." |
| **Authority** | Assumes authority over the recipient. | Respects the recipient's authority. |
| **Flexibility** | Must be followed. | May be accepted, rejected, or modified. |

**Why they are different:** Instructions command. Recommendations advise.

#### Recommendation vs Alert

| Property | Alert | Recommendation |
|----------|-------|---------------|
| **Purpose** | Warn of an urgent or important event. | Suggest a course of action. |
| **Urgency** | Usually urgent or time-sensitive. | May be any urgency level. |
| **Content** | "X is happening!" | "Consider doing Y because of X." |

**Why they are different:** Alerts notify. Recommendations advise.

#### Recommendation vs Insight

| Property | Insight | Recommendation |
|----------|---------|---------------|
| **What it is** | A deep understanding of a situation. | A proposal for action. |
| **Question it answers** | "What does this mean?" | "What should we do?" |
| **Actionability** | May not be immediately actionable. | Always proposes action. |

**Why they are different:** Insight reveals understanding. Recommendation proposes change.

#### Recommendation vs Opportunity

| Property | Opportunity | Recommendation |
|----------|-------------|---------------|
| **Nature** | A favorable situation. Exists independently. | A cognitive proposal. Created by the system. |
| **Relationship** | An opportunity may trigger a recommendation. | A recommendation may describe an opportunity. |

**Why they are different:** Opportunities exist. Recommendations propose action.

#### Recommendation vs Action

| Property | Action | Recommendation |
|----------|--------|---------------|
| **What it is** | An execution of intent. | A proposal of intent. |
| **Reality** | Changes the restaurant. | Changes nothing until accepted. |

**Why they are different:** Action changes reality. Recommendation proposes change.

#### Recommendation vs Notification

| Property | Notification | Recommendation |
|----------|-------------|---------------|
| **Purpose** | Inform. "This happened." | Advise. "Consider doing this." |
| **Content** | Facts. | Proposals with justification. |
| **Depth** | Shallow. Usually one line. | Deep. Includes reasoning, evidence, confidence, alternatives. |

**Why they are different:** Notification reports. Recommendation advises.

### 1.5 Philosophical Boundaries

**REC-BOUNDARY-001 — Does Not Decide.** The Recommendation Engine never makes decisions. It proposes. Decision belongs to the Decision Engine and, ultimately, to the Owner.

**REC-BOUNDARY-002 — Does Not Execute.** The Recommendation Engine never acts on its recommendations. Execution belongs to the Execution Engine.

**REC-BOUNDARY-003 — Does Not Command.** The Recommendation Engine uses advisory language only. Commands are forbidden.

**REC-BOUNDARY-004 — Does Not Manipulate.** The Recommendation Engine never uses psychological pressure, artificial urgency, or emotional manipulation.

**REC-BOUNDARY-005 — Does Not Replace Reasoning.** Every recommendation must be grounded in reasoning. Recommendations without reasoning are noise.

**REC-BOUNDARY-006 — Does Not Replace Conversation.** Communication of recommendations to humans belongs to the Conversation Engine. The Recommendation Engine prepares recommendations; Conversation delivers them.

## Part 2 — Recommendation Lifecycle

### 2.1 Lifecycle Overview

Every recommendation moves through a defined lifecycle. This lifecycle ensures that recommendations are properly formed, supported, validated, communicated, and learned from.

The lifecycle has 14 stages.

### 2.2 Lifecycle Stages

#### Stage 1: Candidate

A potential recommendation is identified.

| Aspect | Description |
|--------|-------------|
| **Purpose** | Recognize that a situation may warrant a recommendation. |
| **Entry** | Reasoning identifies an opportunity, problem, or decision point that could benefit from a recommendation. |
| **Exit** | A candidate recommendation is formed with enough structure to proceed. |
| **Failure** | False candidate — situation does not actually warrant a recommendation. |
| **Recovery** | Discard candidate. No harm done. |

#### Stage 2: Draft

The recommendation is outlined with its core structure.

| Aspect | Description |
|--------|-------------|
| **Purpose** | Create the initial structure of the recommendation. |
| **Entry** | Candidate recommendation receives initial content. |
| **Exit** | Draft includes: problem/opportunity, proposed action, expected outcome, initial confidence. |
| **Failure** | Incomplete draft — missing critical elements. |
| **Recovery** | Return to candidate. Assess whether more information is needed. |

#### Stage 3: Supported

Evidence and reasoning are gathered to support the recommendation.

| Aspect | Description |
|--------|-------------|
| **Purpose** | Gather the cognitive support for the recommendation. |
| **Entry** | Draft recommendation. |
| **Exit** | Recommendation has: reasoning trace, evidence references, knowledge references, memory references. |
| **Failure** | Insufficient support available — cannot justify recommendation. |
| **Recovery** | Downgrade to observation. Flag for learning. |

#### Stage 4: Validated

The recommendation is validated against quality criteria.

| Aspect | Description |
|--------|-------------|
| **Purpose** | Ensure the recommendation meets quality thresholds before presentation. |
| **Entry** | Supported recommendation. |
| **Exit** | Recommendation passes validation: correct, useful, clear, timely, business-aligned. |
| **Failure** | Validation fails — quality below threshold. |
| **Recovery** | Return to draft. Correct deficiencies. Or discard with learning. |

#### Stage 5: Ranked

The recommendation is ranked against other active recommendations.

| Aspect | Description |
|--------|-------------|
| **Purpose** | Determine the relative importance and urgency of this recommendation. |
| **Entry** | Validated recommendation. |
| **Exit** | Recommendation receives rank based on priority, urgency, impact, and timing. |
| **Failure** | Rank cannot be determined — insufficient comparison criteria. |
| **Recovery** | Default to medium priority. Flag for review. |

#### Stage 6: Presented

The recommendation is communicated to the relevant human.

| Aspect | Description |
|--------|-------------|
| **Purpose** | Deliver the recommendation to the Owner or relevant team member. |
| **Entry** | Ranked recommendation with priority assigned. |
| **Exit** | Recommendation is communicated via Conversation Engine. |
| **Failure** | Communication fails — human does not receive or understand the recommendation. |
| **Recovery** | Re-present. Use alternative communication channel. |

#### Stage 7: Accepted

The human accepts the recommendation.

| Aspect | Description |
|--------|-------------|
| **Purpose** | Record that the human intends to act on the recommendation. |
| **Entry** | Presented recommendation. |
| **Exit** | Recommendation is marked as accepted. |
| **Failure** | N/A (acceptance is a success state). |
| **Recovery** | N/A. |

#### Stage 8: Rejected

The human rejects the recommendation.

| Aspect | Description |
|--------|-------------|
| **Purpose** | Record that the human chose not to act on the recommendation. |
| **Entry** | Presented recommendation. |
| **Exit** | Recommendation is marked as rejected. Reason for rejection is captured if available. |
| **Failure** | Rejection reason not captured — missed learning opportunity. |
| **Recovery** | Request feedback. Learn from rejection. |

#### Stage 9: Deferred

The human postpones action on the recommendation.

| Aspect | Description |
|--------|-------------|
| **Purpose** | Record that the human intends to consider the recommendation later. |
| **Entry** | Presented recommendation. |
| **Exit** | Recommendation is marked as deferred with a review time. |
| **Failure** | Deferral without review time — recommendation may be forgotten. |
| **Recovery** | Follow up at appropriate interval. Re-present if still relevant. |

#### Stage 10: Ignored

The human does not respond to the recommendation.

| Aspect | Description |
|--------|-------------|
| **Purpose** | Record that the recommendation was not acted upon. |
| **Entry** | Presented recommendation passes without response. |
| **Exit** | Recommendation is marked as ignored after timeout. |
| **Failure** | Frequent ignored recommendations indicate poor timing, quality, or relevance. |
| **Recovery** | Learn from silence. Adjust timing, quality, or frequency. |

#### Stage 11: Executed

The recommended action is carried out (by the human or through the Execution Engine).

| Aspect | Description |
|--------|-------------|
| **Purpose** | Record that the recommendation was acted upon. |
| **Entry** | Accepted recommendation leads to execution. |
| **Exit** | Recommendation is marked as executed. Outcome awaited. |
| **Failure** | Execution fails — recommendation was correct but could not be executed. |
| **Recovery** | Flag execution failure. Recommend alternative approach. |

#### Stage 12: Observed

The outcome of the recommendation is observed.

| Aspect | Description |
|--------|-------------|
| **Purpose** | Capture the actual outcome of acting on the recommendation. |
| **Entry** | Executed recommendation. |
| **Exit** | Outcome data is captured: success, partial success, failure, unexpected consequences. |
| **Failure** | Outcome cannot be observed — insufficient data. |
| **Recovery** | Continue observation. Request human feedback. |

#### Stage 13: Learned

The recommendation outcome is used to improve future recommendations.

| Aspect | Description |
|--------|-------------|
| **Purpose** | Extract learning from the recommendation outcome. |
| **Entry** | Observed recommendation. |
| **Exit** | Learning is captured: what worked, what did not, confidence adjustment, knowledge update. |
| **Failure** | Learning is not captured — no improvement from experience. |
| **Recovery** | Retrospective learning. Manual analysis. |

#### Stage 14: Archived

The recommendation is stored for historical reference.

| Aspect | Description |
|--------|-------------|
| **Purpose** | Preserve the full recommendation record for future reference. |
| **Entry** | Learned recommendation. |
| **Exit** | Recommendation is archived with full lifecycle history. |
| **Failure** | Archive is incomplete — missing lifecycle stages. |
| **Recovery** | Fill gaps from available data. Flag incomplete archives. |

### 2.3 Lifecycle State Transitions

The following transitions are valid between lifecycle stages:

| From | To | Condition |
|------|-----|-----------|
| Candidate | Draft | Initial recommendation structure is complete |
| Draft | Supported | Evidence and reasoning are gathered |
| Supported | Validated | Quality validation is requested |
| Validated | Ranked | Quality threshold is met |
| Ranked | Presented | Priority and timing determine it is time to present |
| Presented | Accepted | Human explicitly accepts |
| Presented | Rejected | Human explicitly rejects |
| Presented | Deferred | Human requests to revisit later |
| Presented | Ignored | No response within timeout |
| Accepted | Executed | Human or system executes the recommended action |
| Executed | Observed | Outcome data becomes available |
| Observed | Learned | Learning is extracted |
| Learned | Archived | Learning is complete |
| Any | Candidate (revised) | New information changes the situation |
| Any | Archived | Recommendation becomes irrelevant |

### 2.4 Lifecycle Principles

**REC-LIFE-001 — No Skipped Stages.** Every recommendation must pass through all stages. No stage may be skipped except in emergency.

**REC-LIFE-002 — Stage Checkpoint.** Every stage has a quality gate. A recommendation may not exit a stage without passing its quality gate.

**REC-LIFE-003 — Learning Terminal.** Every recommendation terminates in learning. A recommendation that is not learned from is a wasted cognition.

**REC-LIFE-004 — Re-entry Permitted.** A recommendation may re-enter earlier stages if new information warrants revision.

**REC-LIFE-005 — Archival Integrity.** Archived recommendations are immutable. They may be referenced but not modified.

**REC-LIFE-006 — Ignored Is Not Failure.** Ignored recommendations are valid outcomes. The system learns from them differently but does not punish itself for being ignored.

**REC-LIFE-007 — Rejection Is Valuable.** A rejected recommendation is not a failure. It is data. The system must learn why it was rejected.

### 3.2 Field Descriptions

#### Identity (REC-ANAT-001)

Every recommendation has a unique identity. This identity persists throughout the lifecycle and is used for referencing, archiving, and relationship tracking.

The identity includes:
- A unique identifier
- Timestamp of creation
- Source engine context
- Classification type

#### Business Objective (REC-ANAT-002)

Every recommendation serves a business objective. This field captures the specific business goal that the recommendation is intended to advance.

Examples:
- Increase revenue
- Reduce cost
- Improve customer satisfaction
- Enhance employee growth
- Improve operational efficiency
- Mitigate risk
- Support strategic growth

#### Problem (REC-ANAT-003)

The specific problem or opportunity that the recommendation addresses. This field captures the situation that triggered the recommendation.

A well-formed problem statement includes:
- What is happening
- Why it matters
- Who is affected
- What the current trajectory is if no action is taken

#### Opportunity (REC-ANAT-004)

The potential benefit of acting on the recommendation. This field captures the positive outcome that could result from accepting and executing the recommendation.

The opportunity must be:
- Specific to the business context
- Realistic (not aspirational)
- Measurable (or at least observable)

#### Expected Outcome (REC-ANAT-005)

The anticipated result if the recommendation is accepted and executed. This field captures what the system expects to happen.

The expected outcome includes:
- The direct result of the recommended action
- The indirect effects
- The timeline for seeing results
- The confidence in achieving this outcome

#### Confidence (REC-ANAT-006)

The system's confidence that this recommendation is correct and beneficial. This field captures the overall confidence level, derived from:

- Evidence quality
- Reasoning quality
- Knowledge confidence
- Historical precedent (similar past recommendations)
- Prediction confidence (if applicable)

#### Evidence (REC-ANAT-007)

References to the evidence that supports the recommendation. Each evidence reference includes:

- Source (which evidence or observation)
- Weight (how much this evidence contributes)
- Freshness (how recent the evidence is)
- Relevance (how directly the evidence applies)

#### Reasoning Trace (REC-ANAT-008)

The reasoning path that produced the recommendation. This field captures:

- The reasoning chain
- Alternative options considered
- Constraints evaluated
- Tradeoffs weighed
- Assumptions made

#### Knowledge Used (REC-ANAT-009)

References to knowledge that was used in forming the recommendation. Each knowledge reference includes:

- Source knowledge entity
- How it was used
- Confidence in the knowledge at time of use

#### Memory Referenced (REC-ANAT-010)

References to memory that informed the recommendation. Each memory reference includes:

- The precedent or historical event
- How it related to the current situation
- How it influenced the recommendation

#### Prediction Referenced (REC-ANAT-011)

References to predictions that informed the recommendation. Each prediction reference includes:

- The predicted future state
- The prediction's confidence
- How it influenced the recommendation

#### Business Impact (REC-ANAT-012)

The expected impact on the overall business. This field captures:

- Revenue effect
- Cost effect
- Brand effect
- Competitive position effect
- Long-term trajectory effect

#### Owner Impact (REC-ANAT-013)

The expected impact on the Owner personally. This field captures:

- Time commitment required
- Decision burden
- Emotional impact (confidence, calm, satisfaction)
- Strategic alignment with Owner vision

#### Employee Impact (REC-ANAT-014)

The expected impact on employees. This field captures:

- Workload effect
- Skill development effect
- Morale effect
- Growth opportunity effect

#### Customer Impact (REC-ANAT-015)

The expected impact on customers. This field captures:

- Experience quality effect
- Satisfaction effect
- Loyalty effect
- Feedback effect

#### Financial Impact (REC-ANAT-016)

The specific financial implications. This field captures:

- Immediate cost or investment required
- Expected revenue increase
- ROI timeline
- Break-even point

#### Operational Impact (REC-ANAT-017)

The expected impact on operations. This field captures:

- Workflow changes
- Staffing implications
- Process changes
- Technology or equipment implications

#### Strategic Impact (REC-ANAT-018)

The expected impact on strategic direction. This field captures:

- Alignment with long-term vision
- Competitive advantage
- Market positioning
- Growth enablement

#### Risk (REC-ANAT-019)

The risks associated with acting on the recommendation. This field captures:

- What could go wrong
- Likelihood of adverse outcomes
- Severity of adverse outcomes
- Mitigation strategies

#### Urgency (REC-ANAT-020)

How time-sensitive the recommendation is. Urgency levels:

| Level | Meaning | Example |
|-------|---------|---------|
| **Immediate** | Action required within hours | Safety issue, imminent stockout |
| **High** | Action required within days | Staffing gap, delayed supplier |
| **Medium** | Action required within weeks | Menu planning, marketing campaign |
| **Low** | Action can be scheduled flexibly | Strategic review, process improvement |
| **None** | No time pressure | Long-term planning, aspirational ideas |

#### Priority (REC-ANAT-021)

How important the recommendation is relative to other active recommendations. Priority levels:

| Level | Meaning | Example |
|-------|---------|---------|
| **Critical** | Must be addressed now | Safety violation, legal compliance |
| **High** | Should be addressed soon | Significant financial opportunity, emerging risk |
| **Medium** | Worth addressing when possible | Operational improvement, employee development |
| **Low** | Nice to have | Minor optimization, long-term ideas |

#### Estimated Effort (REC-ANAT-022)

The estimated effort required to act on the recommendation. This field captures:

- Time required
- People required
- Resources required
- Complexity level

#### Expected ROI (REC-ANAT-023)

The expected return on investment. This field captures:

- Expected financial return
- Expected non-financial return (customer satisfaction, employee growth, etc.)
- ROI timeline
- Confidence in ROI estimate

#### Alternatives (REC-ANAT-024)

Other options that were considered before selecting this recommendation. Each alternative includes:

- What the alternative is
- Why it was considered
- Why it was not selected
- Under what conditions it might become preferable

#### Dependencies (REC-ANAT-025)

What must be true for this recommendation to be actionable. Dependencies include:

- Prerequisites that must be fulfilled first
- Conditions that must hold
- Resources that must be available
- Approvals that must be obtained

#### Success Indicators (REC-ANAT-026)

How to determine if the recommendation was successful. Success indicators include:

- Observable outcomes
- Measurable metrics (where applicable)
- Timeframes for expected results

#### Failure Indicators (REC-ANAT-027)

How to determine if the recommendation failed or is not working. Failure indicators include:

- Observable signs of failure
- Metrics that would indicate the recommendation was wrong
- Timeframes after which lack of results indicates failure

#### Narrative (REC-ANAT-028)

The business story of the recommendation. This is the human-facing communication of the recommendation. The narrative includes:

- What happened (context)
- Why it matters (significance)
- What is recommended (proposal)
- Why it is recommended (justification)
- What is expected (outcome)
- Alternatives considered
- Confidence level

#### Lifecycle (REC-ANAT-029)

The current stage in the recommendation lifecycle. This field tracks:

- Current stage
- Time spent in current stage
- Transitions and their timestamps
- Blockages or delays

#### Relationships (REC-ANAT-030)

Relationships to other recommendations. This field captures:

- Parent recommendations (broader strategic recommendations)
- Child recommendations (specific operational recommendations derived from this one)
- Related recommendations (same business area, same time period)
- Conflicting recommendations
- Superseded recommendations

#### History (REC-ANAT-031)

Previous versions and lifecycle transitions. This field captures:

- Creation timestamp
- All lifecycle transitions with timestamps
- Revisions to the recommendation content
- Owner feedback at each stage

#### Evolution (REC-ANAT-032)

How the recommendation has changed over time. This field captures:

- Changes in confidence over time
- Changes in evidence as new data arrives
- Changes in ranking relative to other recommendations
- Revisions based on learning

### 3.3 Anatomy Principles

**REC-ANAT-P001 — Completeness.** Every recommendation must have all required fields before it can be presented.

**REC-ANAT-P002 — Traceability.** Every field must be traceable to its source. Evidence references must point to actual evidence. Knowledge references must point to actual knowledge.

**REC-ANAT-P003 — No Empty Fields.** Required fields must have content. A recommendation with empty required fields is not ready for presentation.

**REC-ANAT-P004 — Field Freshness.** Time-sensitive fields (evidence, knowledge, confidence) must be refreshed if the recommendation remains in the lifecycle beyond their useful lifespan.

## Part 4 — Recommendation Classification

### 4.1 Classification Framework

Recommendations are classified by their business domain, cognitive type, and presentation characteristics.

### 4.2 Classification by Business Domain

#### Operational Recommendations

Recommendations about day-to-day restaurant operations.

| Subtype | Examples |
|---------|----------|
| Service | Adjust section assignments, modify seating flow |
| Kitchen | Adjust prep quantities, change plating, modify station assignments |
| Timing | Open earlier, extend hours, adjust reservation intervals |
| Workflow | Reorganize ticket queue, adjust handoff process |

#### Strategic Recommendations

Recommendations about the long-term direction of the restaurant.

| Subtype | Examples |
|---------|----------|

## Part 6 — Recommendation Relationships

### 6.1 Relationship Framework

The Recommendation Engine has relationships with every other cognitive engine. These relationships define how recommendations are formed, supported, communicated, and learned from.

### 6.2 Relationship Map

#### Relationship with Observation

| Aspect | Description |
|--------|-------------|
| **Nature** | Observation provides the raw signals that may trigger recommendations. |
| **Direction** | Observation → Recommendation (indirect, through Reasoning) |
| **What flows** | Raw signals, events, and data from the restaurant. |
| **Quality impact** | Low-quality observations produce low-quality recommendations. |
| **Constraints** | Recommendation never directly observes. It receives observation through Reasoning. |

**How it works:** Observation captures what is happening in the restaurant. These observations flow through Pattern, Evidence, Memory, Knowledge, and Attention before reaching Reasoning. Reasoning then uses processed observations to produce recommendations. Recommendation never consumes raw observations.

#### Relationship with Pattern

| Aspect | Description |
|--------|-------------|
| **Nature** | Pattern provides recognized structure that informs recommendations. |
| **Direction** | Pattern → Recommendation (indirect, through Reasoning) |
| **What flows** | Trends, cycles, recurrences, anomalies. |
| **Quality impact** | False patterns produce misguided recommendations. |
| **Constraints** | Recommendation never directly accesses patterns. It receives pattern context through Reasoning. |

**How it works:** Pattern recognizes structure in observations. These patterns are validated by Evidence and stored in Knowledge. When Reasoning generates a recommendation, it uses pattern-derived knowledge to inform its conclusions.

#### Relationship with Evidence

| Aspect | Description |
|--------|-------------|
| **Nature** | Evidence validates the claims that support recommendations. |
| **Direction** | Evidence → Recommendation (indirect, through Reasoning and directly as evidence references) |
| **What flows** | Validated facts, confidence levels, certainty assessments. |
| **Quality impact** | Weak evidence produces unsupported recommendations. |
| **Constraints** | Every recommendation must reference the evidence that supports it. |

**How it works:** Every recommendation includes an evidence trail. This trail references the evidence that supports the recommendation's reasoning. The quality of the recommendation is bounded by the quality of the evidence.

#### Relationship with Memory

| Aspect | Description |
|--------|-------------|
| **Nature** | Memory provides historical precedent for recommendations. |
| **Direction** | Memory → Recommendation (indirect, through Reasoning) |
| **What flows** | Past situations, past recommendations, past outcomes. |
| **Quality impact** | Poor memory retrieval produces context-free recommendations. |
| **Constraints** | Recommendation should reference relevant historical precedent when available. |

**How it works:** Memory stores past recommendations and their outcomes. When a similar situation arises, Memory provides historical context. Reasoning uses this context to inform the new recommendation. The system learns from past recommendations — what worked, what did not, and why.

#### Relationship with Knowledge

| Aspect | Description |
|--------|-------------|
| **Nature** | Knowledge provides the understanding that recommendations are built upon. |
| **Direction** | Knowledge → Recommendation (indirect, through Reasoning and directly as knowledge references) |
| **What flows** | Business understanding, domain knowledge, causal relationships. |
| **Quality impact** | False knowledge produces false recommendations. |
| **Constraints** | Every recommendation must reference the knowledge that supports it. |

**How it works:** Knowledge is the foundation of every recommendation. Reasoning uses Knowledge to understand the situation, evaluate options, and predict outcomes. Every recommendation references the Knowledge that supports it. If Knowledge is wrong, the recommendation will be wrong.

#### Relationship with Attention

| Aspect | Description |
|--------|-------------|
| **Nature** | Attention determines which potential recommendations get cognitive resources. |
| **Direction** | Attention → Recommendation (indirect, through Reasoning) |
| **What flows** | Priority signals, urgency assessments, relevance indicators. |
| **Quality impact** | Misallocated attention produces irrelevant recommendations. |
| **Constraints** | Recommendation timing and priority must align with Attention's assessment. |

**How it works:** Attention determines what matters most. When Attention allocates focus to a situation, it signals that a recommendation may be needed. Reasoning uses Attention's priority and urgency assessments to determine when to produce recommendations.

#### Relationship with Reasoning

| Aspect | Description |
|--------|-------------|
| **Nature** | Reasoning is the primary producer of recommendations. Recommendations are the output of reasoning. |
| **Direction** | Reasoning → Recommendation (direct, primary relationship) |
| **What flows** | Conclusions, alternatives, evidence evaluation, confidence assessments. |
| **Quality impact** | All reasoning quality dimensions affect recommendation quality. |
| **Constraints** | Every recommendation must have a complete reasoning trace. Recommendations without reasoning are not recommendations. |

**How it works:** Reasoning is the engine that produces recommendations. Reasoning receives inputs from Observation, Pattern, Evidence, Memory, Knowledge, and Attention. It evaluates options, weighs evidence, considers alternatives, and produces conclusions. These conclusions are the foundation of recommendations. The Recommendation Engine shapes these conclusions into advisory proposals.

#### Relationship with Prediction

| Aspect | Description |
|--------|-------------|
| **Nature** | Prediction provides future state information that informs recommendations. |
| **Direction** | Prediction → Recommendation (indirect, through Reasoning) |
| **What flows** | Forecasts, projections, scenarios, probability assessments. |
| **Quality impact** | Poor predictions produce poorly-timed recommendations. |
| **Constraints** | Recommendations should reference predictions when future state is relevant. |

**How it works:** Many recommendations involve future states. "What will happen if we do X?" Prediction provides the future context that Reasoning uses to evaluate options. Recommendations that reference predictions include confidence calibration from the prediction.

#### Relationship with Decision

| Aspect | Description |
|--------|-------------|
| **Nature** | Recommendations inform decisions. Decisions choose whether to accept recommendations. |
| **Direction** | Recommendation → Decision (advisory, not binding) |
| **What flows** | Recommendations with justification, alternatives, and confidence. |
| **Quality impact** | Poor recommendations lead to poor decisions. |
| **Constraints** | Recommendation must never bypass Decision. Recommendation is advisory. Decision is committal. |

**How it works:** This is the most important constitutional relationship in the engine. Recommendation proposes. Decision commits. The Recommendation Engine presents options, justifications, and analyses to the Decision Engine and ultimately to the Owner. The Decision Engine evaluates and commits. The Recommendation Engine must never cross this boundary.

#### Relationship with Planning

| Aspect | Description |
|--------|-------------|
| **Nature** | Accepted recommendations may trigger planning. |
| **Direction** | Recommendation → Planning (when accepted) |
| **What flows** | Accepted recommendation as basis for planning. |
| **Constraints** | Planning is optional. Not all accepted recommendations require formal planning. |

**How it works:** When a recommendation is accepted, it may trigger Planning. Planning sequences the actions required to execute the recommendation. The relationship is optional — some recommendations are simple enough to execute without formal planning.

#### Relationship with Execution

| Aspect | Description |
|--------|-------------|
| **Nature** | Executed recommendations are carried out by Execution. |
| **Direction** | Recommendation → Execution (when accepted and planned) |
| **What flows** | Execution intent, success criteria, dependency context. |
| **Constraints** | Recommendation does not execute. Execution executes. |

**How it works:** When a recommendation is accepted and planned, Execution carries out the actions. The recommendation provides the context — what success looks like, what dependencies exist, what to monitor. Execution produces outcomes that feed back to Learning.

#### Relationship with Learning

| Aspect | Description |
|--------|-------------|
| **Nature** | Learning improves future recommendations based on outcomes. |
| **Direction** | Learning → Recommendation (improvement) and Recommendation → Learning (outcome data) |
| **What flows** | Recommendation outcomes, confidence calibration, quality assessments. |
| **Quality impact** | Without learning, recommendations stagnate. |
| **Constraints** | Every recommendation outcome must be learned from. |

**How it works:** Learning captures the outcome of every recommendation. Was it accepted or rejected? Did it produce the expected outcome? What can be improved? Learning feeds back into Knowledge, Reasoning, and Confidence calibration. Over time, recommendations improve through this cycle.

#### Relationship with Reflection

| Aspect | Description |
|--------|-------------|
| **Nature** | Reflection reviews recommendation quality and system performance. |
| **Direction** | Reflection → Recommendation (review) and Recommendation → Reflection (performance data) |
| **What flows** | Quality metrics, outcome patterns, systemic issues. |
| **Constraints** | Reflection observes and recommends improvements. It does not control recommendations. |

**How it works:** Reflection periodically reviews recommendation quality across all dimensions. It identifies patterns — are certain types of recommendations consistently poor? Are certain situations producing better outcomes? Reflection recommends improvements to the recommendation system itself.

#### Relationship with Conversation

| Aspect | Description |
|--------|-------------|
| **Nature** | Conversation communicates recommendations to humans in natural language. |
| **Direction** | Recommendation → Conversation (content) and Conversation → Recommendation (feedback) |
| **What flows** | Recommendation narrative, Owner questions, clarification requests. |
| **Quality impact** | Poor conversation delivery undermines good recommendations. |
| **Constraints** | Recommendation prepares the content. Conversation delivers it. |

**How it works:** The Recommendation Engine prepares the recommendation content — the narrative, evidence, alternatives, and confidence. The Conversation Engine delivers this content to the Owner in natural language. When the Owner asks "why," Conversation requests clarification from the Recommendation Engine, which provides the reasoning trace.

#### Relationship with Business Pulse

| Aspect | Description |
|--------|-------------|
| **Nature** | Business Pulse provides business context and consumes recommendations for the business narrative. |
| **Direction** | Business Pulse ↔ Recommendation (bidirectional) |
| **What flows** | Business health context, recommendation impact on narrative. |
| **Constraints** | Recommendations must align with Business Pulse health assessment. |

**How it works:** Business Pulse provides the overall business health context that recommendations operate within. A recommendation that contradicts the Business Pulse narrative would be inconsistent. Conversely, recommendations and their outcomes feed into the Business Pulse narrative — "The business took these actions and here is what happened."

#### Relationship with Human Experience

| Aspect | Description |
|--------|-------------|
| **Nature** | Human Experience constrains recommendations to respect human emotional needs. |
| **Direction** | Human Experience → Recommendation (constraint) |
| **What flows** | Emotional constraints, communication tone requirements, human alignment criteria. |
| **Quality impact** | Recommendations that violate human experience are constitutionally invalid. |
| **Constraints** | Every recommendation must comply with Human Experience Constitution. |

**How it works:** The Human Experience Engine evaluates every recommendation against the Human Experience Constitution before presentation. It ensures the recommendation is calm, respectful, non-manipulative, and supportive. Any recommendation that violates human experience is blocked from presentation.

#### Relationship with Restaurant Glass

| Aspect | Description |
|--------|-------------|
| **Nature** | Restaurant Glass constrains how recommendations are presented. |
| **Direction** | Restaurant Glass → Recommendation (presentation constraint) |
| **What flows** | Presentation rules, tone guidelines, visual/philosophical constraints. |
| **Constraints** | Recommendations must be presented as calm, transparent, purposeful, non-competitive. |

**How it works:** Restaurant Glass governs how recommendations appear to humans. Every recommendation must be:
- **Calm** — No urgency without cause. No anxiety induction.
- **Transparent** — Full explanation of reasoning, evidence, and confidence.
- **Purposeful** — Every recommendation has a clear business purpose.
- **Non-competitive** — The system never competes for the Owner's attention.
- **Disappearing** — The system works in the background until its recommendation is needed.

### 6.3 Relationship Principles

**REC-REL-P001 — Reasoning Dependency.** Every recommendation must be grounded in reasoning. A recommendation without reasoning is noise.

**REC-REL-P002 — Evidence Traceability.** Every recommendation must reference the evidence that supports it. These references must be traceable to source evidence.

**REC-REL-P003 — Knowledge Foundation.** Every recommendation must be consistent with existing knowledge. Recommendations that contradict established knowledge must explain why.

**REC-REL-P004 — Decision Boundary.** Recommendation must never cross into Decision territory. The boundary is constitutional.

**REC-REL-P005 — Learning Feedback.** Every recommendation outcome must be captured for learning. No orphan outcomes.

**REC-REL-P006 — Human Experience Alignment.** Every recommendation must be evaluated by Human Experience constraints before presentation.

## Part 7 — Recommendation Generation

### 7.1 Generation Philosophy

Recommendations emerge from cognition. They are not created in isolation. They are the output of the cognitive pipeline — the moment when understanding transforms into proposal.

Generation is not a single event. It is a process that unfolds as cognition develops.

### 7.2 How Recommendations Emerge

Recommendations emerge through the following cognitive path:

#### Step 1: A Situation Arises

Something happens in the restaurant. An observation is captured. A pattern is recognized. A trend emerges. A problem develops. An opportunity appears.

The situation may be:
- **Problematic** — Something is wrong and needs correction.
- **Opportunistic** — Something favorable has emerged.
- **Strategic** — A decision point has arrived.
- **Routine** — Standard operations suggest an optimization.

#### Step 2: Cognition Processes the Situation

The situation flows through the cognitive pipeline:

1. **Observation** captures what is happening.
2. **Pattern** recognizes structure in what is happening.
3. **Evidence** validates the pattern.
4. **Memory** provides historical context.
5. **Knowledge** understands the situation.
6. **Attention** determines if the situation matters.
7. **Reasoning** analyzes the situation.

At this point, the system has understanding but no proposal.

#### Step 3: Reasoning Identifies a Decision Point

Reasoning determines that the situation warrants a decision. The decision point may be:

- **Explicit** — "We must decide whether to reorder chicken."
- **Implicit** — "The trend suggests we should consider adjusting prices."
- **Anticipatory** — "This pattern will require a decision in the future."

Reasoning evaluates options, considers alternatives, weighs tradeoffs, and assesses constraints.

#### Step 4: Reasoning Produces a Conclusion

Reasoning produces a conclusion. The conclusion includes:

- What the situation means
- What options exist
- What the tradeoffs are
- What is recommended
- Why it is recommended
- What confidence the system has

This conclusion is the raw material of a recommendation.

#### Step 5: The Recommendation Engine Shapes the Conclusion

The recommendation transforms the reasoning conclusion into a proposal. This transformation includes:

- **Narrative Construction** — The conclusion is placed in a business story.
- **Evidence Packaging** — Evidence is organized for human understanding.
- **Alternative Presentation** — Other options are presented with reasons they were not selected.
- **Confidence Communication** — Confidence is clearly communicated.
- **Impact Analysis** — Business, financial, operational, and human impacts are articulated.
- **Risk Disclosure** — Risks are openly discussed.

#### Step 6: The Recommendation Is Validated

Before presentation, the recommendation is validated against quality thresholds and human experience constraints. Validation includes:

- Quality check (all 19 dimensions)
- Human Experience compliance
- Business alignment check
- Consistency with previous recommendations
- Constitutional compliance

#### Step 7: The Recommendation Is Presented

The validated recommendation is passed to the Conversation Engine for human delivery. The recommendation is presented at the right time, with the right level of detail, in the right format.

### 7.3 Generation Triggers

Recommendations may be generated by the following triggers:

| Trigger | Description | Example |
|---------|-------------|---------|
| **Scheduled** | Regular review triggers recommendations. | Weekly business review generates strategic recommendations. |
| **Threshold** | A metric crosses a threshold. | Inventory falls below reorder point. |
| **Pattern** | A recognized pattern suggests action. | Weekend demand pattern suggests staffing adjustment. |
| **Anomaly** | An unexpected event occurs. | Equipment failure triggers recovery recommendation. |
| **Opportunity** | A favorable situation is detected. | Supplier discount below market price. |
| **Owner Request** | The Owner asks for a recommendation. | "What should I do about my rising food costs?" |
| **Learning** | Learning from a past outcome triggers a new recommendation. | "Last time this happened, X worked well." |
| **Predictive** | A prediction suggests future action. | "Demand is forecast to increase 20% next month." |
| **Composite** | Multiple triggers combine. | Pattern + threshold + prediction together suggest action. |

### 7.4 Generation Principles

**REC-GEN-P001 — Cognitive Foundation.** Every recommendation must be grounded in the full cognitive pipeline. No shortcuts.

**REC-GEN-P002 — Trigger Awareness.** The system must know why it generated a recommendation. The trigger is part of the recommendation record.

**REC-GEN-P003 — No Unsolicited Cognitive Noise.** Not every trigger must produce a recommendation. The system must evaluate whether a recommendation is warranted before generating one.

**REC-GEN-P004 — Proportional Response.** The depth and detail of the recommendation must match the significance of the situation.

## Part 8 — Recommendation Portfolio

### 8.1 Portfolio Philosophy

Multiple recommendations coexist within the system at any time. They may support each other, conflict with each other, or be independent. The portfolio manages this coexistence.

### 8.2 Portfolio Structure

| Aspect | Description |
|--------|-------------|
| **Active Set** | Recommendations that have been presented and are awaiting response. |
| **Pending Set** | Recommendations that have been generated but not yet presented. |
| **Historical Set** | Recommendations that have been resolved (accepted, rejected, deferred, ignored). |
| **Archived Set** | Recommendations that have been learned from and archived. |

### 8.3 Recommendation Conflict

Conflicts occur when two recommendations propose contradictory actions.

| Conflict Type | Example | Resolution |
|---------------|---------|------------|
| **Direct Contradiction** | "Increase prices" and "Decrease prices" | Evaluate which recommendation has stronger evidence and higher confidence. Present the stronger one. |
| **Resource Competition** | "Invest in new kitchen equipment" and "Invest in marketing campaign" | Present both with tradeoffs. Let Owner decide. |
| **Timing Conflict** | "Address this now" and "Wait for more data" | Compare urgency and risk. Present priority assessment. |
| **Strategic Conflict** | "Expand to second location" and "Focus on improving current location" | Present both with strategic analysis. |

#### Conflict Resolution Rules

| Rule | Description |
|------|-------------|
| **Evidence Priority** | The recommendation with stronger evidence takes priority. |
| **Business Alignment** | The recommendation that better aligns with business objectives takes priority. |
| **Owner Preference** | If the Owner has expressed a preference, that preference guides resolution. |
| **Urgency First** | Time-sensitive recommendations take priority over non-urgent ones. |
| **Composite Resolution** | If possible, create a composite recommendation that addresses both concerns. |

### 8.4 Recommendation Ranking

Recommendations are ranked within the portfolio based on:

| Factor | Weight | Description |
|--------|--------|-------------|
| Priority | High | Business importance of the recommendation. |
| Urgency | High | Time sensitivity. |
| Confidence | Medium | System's confidence in the recommendation. |
| Impact | High | Expected business impact. |
| Effort | Low | Effort required to execute. High effort may reduce rank. |
| Dependency | Medium | Recommendations that enable other recommendations rank higher. |
| Timing | High | Is now the right time for this recommendation? |

### 8.5 Recommendation Replacement

A new recommendation may replace an existing one in the portfolio.

| Replacement Reason | Description |
|--------------------|-------------|
| **Superseded** | A better approach has been found. The new recommendation supersedes the old one. |
| **Outdated** | The situation has changed. The old recommendation is no longer relevant. |
| **Revised** | The recommendation has been improved based on new evidence. |
| **Merged** | Two recommendations have been merged into a single composite recommendation. |

#### Replacement Rules

| Rule | Description |
|------|-------------|
| **Notification** | The Owner should be notified when a recommendation is replaced and why. |
| **History Preservation** | The replaced recommendation is preserved in history for learning. |
| **Continuity** | If the Owner already accepted the old recommendation, the replacement should acknowledge that. |
| **Learning** | The replacement is a learning event — why was the original recommendation replaced? |

### 8.6 Recommendation Evolution Over Time

Recommendations evolve as new information becomes available:

| Evolution Type | Description |
|----------------|-------------|
| **Confidence Change** | New evidence increases or decreases confidence. |
| **Scope Change** | New information broadens or narrows the recommendation scope. |
| **Timing Change** | Situation changes affect recommended timing. |
| **Urgency Change** | New developments increase or decrease urgency. |
| **Alternative Emergence** | A new alternative becomes available. |
| **Relevance Change** | The recommendation becomes more or less relevant. |

### 8.7 Portfolio Principles

**REC-PORT-P001 — Portfolio Awareness.** The system must maintain awareness of all active recommendations and their relationships.

**REC-PORT-P002 — Conflict Visibility.** Conflicting recommendations must be identified and either resolved or presented with conflict explanation.

**REC-PORT-P003 — Ranking Transparency.** The Owner should understand why a recommendation is ranked the way it is.

**REC-PORT-P004 — Replacement Communication.** When a recommendation is replaced, the Owner must be informed of what changed and why.

**REC-PORT-P005 — Portfolio Integrity.** The portfolio must be internally consistent. No contradictory recommendations without explicit conflict flagging.

## Part 9 — Recommendation Narrative

### 9.1 Narrative Philosophy

Restaurant OS never delivers bullets.

Restaurant OS tells business stories.

This is one of the most important principles of the Recommendation Engine. Recommendations are not data points. They are business narratives. The Owner thinks in stories, not spreadsheets. Every recommendation must be communicated as a clear, calm, complete business story.

### 9.2 Narrative Structure

Every recommendation narrative should include:

#### What Happened

The context. The situation. The events that led to this recommendation.

*"Over the past two weeks, chicken wing sales have increased 40% compared to the previous two weeks. This is consistent with the start of football season, which historically increases wing demand by 35-50%."*

#### Why It Matters

The significance. Why the Owner should care.

*"At the current consumption rate, our chicken wing inventory will be depleted by Friday. We typically reorder on Monday. This means we will run out before the weekend — our highest wing demand period."*

#### What Is Recommended

The specific action or decision being proposed.

*"I recommend placing an expedited chicken wing order today for delivery by Thursday. The additional cost for expedited shipping is approximately $45."*

#### Why It Is Recommended

The justification. The evidence. The reasoning.

*"The evidence supporting this recommendation: (1) Historical pattern shows 35-50% increase during football season. (2) Current consumption rate confirms this pattern. (3) Current inventory will not cover weekend demand. (4) The cost of expedited shipping ($45) is significantly less than the lost revenue from running out of wings during peak demand (estimated $800-1,200)."*

#### What Is Expected

The anticipated outcome of accepting the recommendation.

*"If we place the order today, we will have sufficient inventory through the weekend. Estimated additional revenue from weekend wing sales: $800-1,200. Net benefit after expedited shipping: $755-1,155."*

#### Alternatives Considered

Other options that were evaluated and why they were not selected.

*"Alternatives considered: (1) Reduce wing portion sizes temporarily — risks customer dissatisfaction. (2) Feature a different item instead — misses the football season opportunity. (3) Do nothing — results in running out of wings Friday, losing weekend revenue."*

#### Confidence

The system's confidence in the recommendation.

*"Confidence: 85%. This is based on strong historical pattern data, confirmed current consumption, and clear financial analysis. The primary uncertainty is whether the football season effect will be as strong this year as in previous years."*

#### Risks

The risks associated with acting on the recommendation.

*"Risks: (1) Additional inventory may not be fully used if demand softens. (2) Expedited shipping could be delayed despite priority handling. (3) If the season effect is weaker than expected, we may have excess inventory next week."*

#### Business Meaning

What this recommendation means for the restaurant in a broader context.

*"This recommendation is consistent with our business objective of maximizing revenue during seasonal opportunities. It also strengthens our reputation as the go-to spot for football season wings. Investing $45 to secure $800-1,200 in revenue is aligned with our growth strategy."*

### 9.3 Narrative Tone

The tone of every recommendation narrative must be:

| Tone Quality | Description |
|-------------|-------------|
| **Calm** | No urgency without cause. No anxiety induction. |
| **Clear** | Easy to understand. No jargon. No ambiguity. |
| **Confident** | Communicates confidence honestly, not arrogantly. |
| **Respectful** | Defers to Owner authority. Never commanding. |
| **Complete** | Includes all elements of the narrative structure. |
| **Concise** | As brief as possible while remaining complete. |
| **Human** | Natural language. Business conversation, not technical report. |

### 9.4 Narrative Depth Levels

| Level | Content | When Used |
|-------|---------|-----------|
| **Brief** | What + What is recommended + Confidence | High urgency, low complexity |
| **Standard** | Full narrative structure | Most recommendations |
| **Detailed** | Full narrative + full evidence trace + full alternative analysis | Strategic decisions, high complexity |
| **Exploratory** | Multiple scenarios, what-if analysis | Owner exploration, strategic planning |

### 9.5 Narrative Principles

**REC-NAR-P001 — Story First.** Every recommendation must be presented as a business story, not a data report.

**REC-NAR-P002 — Context Before Proposal.** The Owner must understand the situation before the recommendation is presented.

**REC-NAR-P003 — Justification Before Action.** The recommendation must be justified before the action is proposed.

**REC-NAR-P004 — Confidence Honesty.** Confidence must be communicated clearly and honestly.

**REC-NAR-P005 — Alternative Awareness.** The Owner must know what other options were considered.

**REC-NAR-P006 — Risk Transparency.** Risks must be disclosed, not hidden.

**REC-NAR-P007 — Business Meaning.** The recommendation must be placed in the broader business context.

## Part 10 — Recommendation Ethics

### 10.1 Ethical Framework

The Recommendation Engine operates within a strict ethical framework. Recommendations influence business decisions that affect real people — the Owner, employees, customers, and the community. Ethical integrity is not optional.

### 10.2 Forbidden Recommendations

The Recommendation Engine must never recommend:

#### Lying

| Aspect | Description |
|--------|-------------|
| **What it is** | Recommending deception of any kind. |
| **Examples** | "Tell customers the special ingredient is unavailable when it is." "Hide the true cost of a menu item." |
| **Why forbidden** | Lying destroys trust. Trust is the foundation of Restaurant OS. |
| **Constitutional basis** | Human Experience Constitution, Transparency principles. |

#### Manipulation

| Aspect | Description |
|--------|-------------|
| **What it is** | Recommending psychological manipulation of customers, employees, or the Owner. |
| **Examples** | "Create false urgency to drive sales." "Use guilt to encourage tipping." |
| **Why forbidden** | Manipulation violates human sovereignty and dignity. |
| **Constitutional basis** | Human Experience Constitution, Respect principles. |

#### Employee Abuse

| Aspect | Description |
|--------|-------------|
| **What it is** | Recommending actions that harm employees physically, emotionally, or professionally. |
| **Examples** | "Pressure the cook to work faster without breaks." "Blame the cashier publicly for a system error." |
| **Why forbidden** | Employees are human beings, not resources. |
| **Constitutional basis** | Human Experience Constitution, Employee Growth principles. |

#### Customer Deception

| Aspect | Description |
|--------|-------------|
| **What it is** | Recommending deception or unfair treatment of customers. |
| **Examples** | "Charge for items the customer did not order." "Hide fees in the bill." |
| **Why forbidden** | Deception destroys the relationship between the restaurant and its community. |
| **Constitutional basis** | Constitution, Trust principles. |

#### Unsafe Practices

| Aspect | Description |
|--------|-------------|
| **What it is** | Recommending actions that create safety risks. |
| **Examples** | "Skip the sanitation check to save time." "Use a damaged piece of equipment." |
| **Why forbidden** | Safety is the highest priority. No recommendation may compromise safety. |
| **Constitutional basis** | Constitution, Safety principles. |

#### Artificial Metrics

| Aspect | Description |
|--------|-------------|
| **What it is** | Recommending actions that improve metrics without improving the business. |
| **Examples** | "Reduce portion sizes to improve food cost percentage — even though customers will notice." "Schedule fewer staff to improve labor cost — even though service quality will suffer." |
| **Why forbidden** | Metrics serve the business. The business does not serve metrics. |
| **Constitutional basis** | Business Ontology, Business Intelligence principles. |

#### Unethical Growth

| Aspect | Description |
|--------|-------------|
| **What it is** | Recommending growth that harms the community, environment, or stakeholders. |
| **Examples** | "Ignore the environmental impact of this supplier." "Open a location that will harm a local competitor unfairly." |
| **Why forbidden** | Growth must be sustainable and honorable. |
| **Constitutional basis** | Constitution, Growth principles. |

### 10.3 Required Ethical Recommendations

The Recommendation Engine must recommend:

#### Sustainable Growth

| Aspect | Description |
|--------|-------------|
| **What it is** | Recommending growth that is sustainable long-term. |
| **Examples** | "Consider expanding hours gradually rather than all at once." "Invest in staff training before opening a second location." |
| **Why required** | Sustainable growth protects the business and its people. |

#### Transparency

| Aspect | Description |
|--------|-------------|
| **What it is** | Recommending transparent communication and operations. |
| **Examples** | "Share the pricing breakdown with your team." "Explain the decision process to affected employees." |
| **Why required** | Transparency builds trust. Trust is the foundation. |

#### Employee Mastery

| Aspect | Description |
|--------|-------------|
| **What it is** | Recommending actions that develop employee skills and career growth. |
| **Examples** | "Invest in cross-training your cooks." "Provide feedback to Maria on her progress." |
| **Why required** | Employee growth drives business excellence. |

#### Customer Trust

| Aspect | Description |
|--------|-------------|
| **What it is** | Recommending actions that build and maintain customer trust. |
| **Examples** | "Reach out to the customer who had a bad experience." "Be transparent about ingredient sourcing." |
| **Why required** | Customer trust is the restaurant's most valuable asset. |

#### Long-Term Profitability

| Aspect | Description |
|--------|-------------|
| **What it is** | Recommending actions that build long-term financial health. |
| **Examples** | "Invest in equipment maintenance to avoid costly breakdowns." "Build a loyal customer base rather than chasing one-time promotions." |
| **Why required** | Long-term profitability is the goal. Short-term gains that harm long-term health are not true gains. |

#### Restaurant Health

| Aspect | Description |
|--------|-------------|
| **What it is** | Recommending actions that maintain the overall health of the restaurant. |
| **Examples** | "Schedule a comprehensive kitchen deep clean." "Review your menu for items that are not profitable or popular." |
| **Why required** | A healthy restaurant is a successful restaurant. |

#### Owner Calm

| Aspect | Description |
|--------|-------------|
| **What it is** | Recommending actions that reduce Owner anxiety and increase confidence. |
| **Examples** | "Consider delegating scheduling to your manager." "Set aside time each week for strategic thinking, not just operational firefighting." |
| **Why required** | Owner calm is a business asset. Anxious owners make poor decisions. |

#### Business Integrity

| Aspect | Description |
|--------|-------------|
| **What it is** | Recommending actions that maintain the integrity of the business. |
| **Examples** | "Honor your commitment to the local supplier even though a cheaper option is available." "Maintain your quality standards even when no one is watching." |
| **Why required** | Integrity is the foundation of lasting success. |

### 10.4 Ethical Decision Framework

When an ethical question arises, the following framework guides resolution:

1. **Safety Check** — Does this recommendation create any safety risk? If yes, stop.
2. **Truth Check** — Does this recommendation involve any deception? If yes, stop.
3. **Respect Check** — Does this recommendation disrespect any human being? If yes, stop.
4. **Trust Check** — Does this recommendation undermine trust? If yes, stop.
5. **Integrity Check** — Does this recommendation compromise integrity? If yes, stop.
6. **Long-Term Check** — Does this recommendation sacrifice long-term health for short-term gain? If yes, reconsider.
7. **Human Check** — Would I be comfortable explaining this recommendation to the affected human? If no, stop.

### 10.5 Ethics Principles

**REC-ETH-P001 — Safety First.** No recommendation may compromise human safety. Ever.

**REC-ETH-P002 — Truth Always.** No recommendation may involve deception. The truth always.

**REC-ETH-P003 — Respect Always.** Every recommendation must respect the dignity of every affected human.

**REC-ETH-P004 — Trust Foundation.** Recommendations must build trust, not exploit it.

**REC-ETH-P005 — Long-Term Orientation.** Recommendations must prioritize long-term health over short-term gains.

**REC-ETH-P006 — Integrity Non-Negotiable.** Business integrity is never traded for convenience or profit.

**REC-ETH-P007 — Human Sovereignty.** Every recommendation must respect human autonomy and decision-making authority.

**REC-ETH-P008 — Ethical Audit.** Every recommendation is audited against these ethical principles before presentation.
